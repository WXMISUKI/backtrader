## 1. 背景

Phase 72 已经把东方财富历史行情失败收束成结构化语义，明确了 `failure_kind / failure_stage / failure_code / fallback_strategy`。

但当前问题还没有真正解决到“可稳定投产”的程度，因为项目仍然会遇到两类现实约束：

1. 单一历史数据源仍可能被远端断开或限流
2. cookie 需要人工刷新，且刷新动作目前依赖手工打开浏览器、进入页面、查看存储并回填

这意味着系统虽然能解释失败，但还不能在失败时自动选择更可用的数据源，也不能把 cookie 刷新收成一条更顺手的本地自动化链路。

因此，本阶段的重点不是继续优化诊断文案，而是把“失败可解释”推进到“失败可绕行、cookie 可刷新、数据源可替换”。

---

## 2. 目标

### 2.1 总目标

建立一个最小但足够实用的历史行情多源路由闭环，并把 cookie 刷新收束成可复用的本地自动化步骤。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 多源可选 | 支持至少 2 个历史行情来源候选，避免单源失效就全链路降级 |
| 路由可见 | 明确记录每次请求尝试了哪个 provider、失败在什么阶段、最终选了谁 |
| 兜底可控 | 最终 fallback 仍然显式标记，不把 mock 混成真实数据 |
| cookie 自动化 | 把手工刷新 cookie 的重复步骤收成可复用 helper/skill |
| 低侵入 | 不重写主业务链路，不引入复杂编排平台 |
| 兼容现状 | 保留 Phase 72 的结构化失败语义和现有治理快照 |

---

## 3. 范围

### 3.1 纳入范围

- 历史行情 provider 契约抽象
- provider 可用性预检
- provider 路由和回退策略
- 治理快照记录 provider 尝试链
- cookie 刷新 helper / skill 设计
- 联调脚本与日常摘要的多源展示
- 回归测试和运行说明

### 3.2 不纳入范围

- 自动交易
- 复杂代理池
- 大规模异步重试框架
- 付费数据源接入
- 需要人工登录验证码规避的自动化

---

## 4. 设计原则

### 4.1 先保证真实数据可绕行

当一个 provider 不稳定时，系统应先尝试下一个可用真实来源，而不是立刻落到 mock。

### 4.2 cookie 只做本地加速，不做权限绕过

cookie 自动化仅用于把用户已经合法登录后的会话状态转成可复用配置，不做任何权限绕过。

### 4.3 路由优先于修补

当前阶段不再围着单一接口堆修补，而是把数据源切换、失败识别和兜底行为统一纳入路由层。

### 4.4 保持可解释

每次请求最终使用了哪个 provider、为什么没选别的 provider、mock 是不是兜底，都必须显式可见。

## 5. 候选数据源

### 5.1 建议的候选顺序

| 优先级 | Provider | 角色 | 说明 |
|---|---|---|---|
| P0 | `eastmoney_direct` | 主候选 | 继续保留，作为当前最常用的站点级来源 |
| P0 | `akshare_hist` | 备选主源 | 通过 AKShare 提供统一封装后的历史行情 |
| P1 | `ths_hist` | 次级候选 | 作为同花顺相关的站点级来源候选，是否可用以本地验证为准 |
| P2 | `offline_mock` | 最终兜底 | 仅用于离线演示、回归和保护上层流程 |

### 5.2 候选选择原则

- 优先真实数据
- 优先可重复、可自动化、可回归的来源
- 备用源失败时仍要保留失败原因
- mock 只能在明确无法获得真实数据时使用

## 6. cookie 刷新自动化

### 6.1 目标

把目前需要手工完成的 cookie 刷新动作收成一个本地可复用步骤，降低日常维护成本。

### 6.2 支持形式

建议优先按这两个层次推进：

1. 本地 helper 脚本
2. 可选 browser skill / browser automation flow

### 6.3 自动化边界

允许做的事情：

- 打开目标站点
- 读取当前浏览器会话中的 cookie
- 导出为本地文本或 JSON
- 更新本地配置文件
- 立刻触发历史行情 smoke test

不允许做的事情：

- 绕过验证码
- 自动提交登录凭据
- 自动复制敏感内容到外部服务

### 6.4 推荐交互模式

日常使用时，用户只需：

1. 打开已经登录好的浏览器
2. 运行本地 cookie refresh helper
3. 让 helper 自动抓取目标域 cookie
4. 立即运行历史行情验证

## 7. 输出约定

### 7.1 provider 路由输出

建议在治理快照中补充：

- `selected_provider`
- `provider_attempts`
- `provider_status`
- `provider_reason`
- `provider_latency_ms`
- `fallback_strategy`

### 7.2 provider_attempts 结构

每次尝试建议记录：

- `provider`
- `status`
- `failure_stage`
- `failure_code`
- `reason`
- `rows`
- `latency_ms`

### 7.3 cookie 自动化输出

cookie helper 建议输出：

- `cookie_source`
- `cookie_loaded`
- `cookie_domain`
- `cookie_keys`
- `has_session_cookie`
- `last_refreshed_at`
- `refresh_status`

---

## 8. 验收标准

- 历史行情失败后，系统可以尝试下一个 provider，而不是直接进入 mock
- 最终选择的 provider 在治理快照中可见
- 失败的 provider 尝试链能被联调脚本看见
- cookie 刷新可被本地 helper 自动完成，不再依赖每次人工拼接
- mock 兜底依然保留，但不再是默认首选
- 现有 Phase 72 的失败分层不被破坏
