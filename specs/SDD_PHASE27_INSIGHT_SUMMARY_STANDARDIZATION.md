## 1. 背景

当前项目已经具备：

- 统一决策摘要收紧
- 最小反馈统计入口
- 最小反馈洞察入口

但是真实使用时，用户通常不只看 `/decision`，还会看：

- `/decision/stats`
- `/decision/insights`

如果这几个入口的表达方式不一致，用户每次都要重新理解一遍字段结构，会增加使用成本。为了让系统更适合“快速投产、真实测试、边用边改”，本阶段要把统计与洞察输出也收紧为一致的阅读模板。

---

## 2. 目标

### 2.1 总目标

让 `/decision/stats` 和 `/decision/insights` 也具备统一、直接、可阅读的摘要卡片。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 一致阅读 | 统计与洞察也能快速看到结论、依据、风险、下一步动作 |
| 轻量统一 | 不新增复杂报表，只增加稳定摘要卡片 |
| 兼容现有 | 保留原有 `data` 与统计结构 |
| 方便联调 | 示例脚本直接能打印摘要卡片 |
| 支持迭代 | 后续如果继续优化，先看摘要卡片即可 |

---

## 3. 范围

### 3.1 纳入范围

- `StockOrchestrator.get_decision_session_stats`
- `StockOrchestrator.get_decision_feedback_insights`
- `core/agent/session.py` 的统计输出
- `examples/api_demo.py` 的统计 / 洞察展示
- `README_QUANT.md` 的使用说明

### 3.2 不纳入范围

- 新增 `/decision/overview` 这类新接口
- 新数据库或新 BI 页面
- 自动调整路由或策略
- 复杂图表与可视化面板

---

## 4. 设计原则

### 4.1 先看摘要，再看细节

统计与洞察也要优先给出四段式摘要，减少阅读成本。

### 4.2 保留原结构

原有 `data`、`meta` 和统计明细继续保留，避免破坏调用方。

### 4.3 只做表达收口

本阶段不引入新推理，只把已有统计与洞察结果包装得更好读。

### 4.4 样本门槛提示

统计与洞察都要提醒样本量是否足够，避免过度解读。

## 5. 接口约定

### 5.1 `GET /decision/stats`

建议增加：

- `stats_summary`
- `结论`
- `依据`
- `风险`
- `下一步动作`
- `conclusion`
- `basis`
- `risk`
- `next_action`

### 5.2 `GET /decision/insights`

建议增加：

- `insights_summary`
- `结论`
- `依据`
- `风险`
- `下一步动作`
- `conclusion`
- `basis`
- `risk`
- `next_action`

### 5.3 输出要求

输出至少应保持：

- `ok`
- `tool`
- `category`
- `data_source`
- `summary`
- `data`
- `meta`

并在顶层附加一个可直接阅读的摘要卡片。

---

## 6. 验收标准

- `/decision/stats` 能直接展示摘要卡片
- `/decision/insights` 能直接展示摘要卡片
- 原有统计与洞察明细仍然保留
- `examples/api_demo.py` 先显示摘要卡片，再显示 JSON 细节
- 本地无需新增部署依赖即可验证
