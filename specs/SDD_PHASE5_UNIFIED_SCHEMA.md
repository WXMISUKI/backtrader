## 1. 背景

当前项目已经拥有分析、推荐、市场概览、风控、回测、报告等能力。但不同工具的输出风格仍存在差异：

- 有的直接返回业务对象
- 有的返回字典
- 有的已经带治理信息
- 有的还只是简单 mock 包装

这会导致智能体在做工具调用和结果总结时，仍然需要“理解每个工具的私有形状”。

为了让智能体真正稳定接入，需要把所有业务级工具统一成同一输出 schema。

---

## 2. 目标

### 2.1 总目标

建立一套适用于智能体调用的统一输出 schema，让所有业务级工具都能用同一结构返回结果。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 统一信封 | 所有工具返回 `ok / tool / category / data_source / summary / data / meta` |
| 统一治理 | 所有工具都要带 `governance` 或可推导治理信息 |
| 统一语义 | `real / mock / cache / config / unknown` 语义一致 |
| 统一可读性 | 智能体可直接总结，无需理解每个对象的内部实现 |
| 兼容旧逻辑 | 现有业务对象仍可被序列化和消费 |

---

## 3. 范围

### 3.1 纳入范围

- 个股分析
- 基本面分析
- 市场概览
- 风控配置
- 回测结果
- 推荐结果
- 报告生成

### 3.2 不纳入范围

- 工具内部算法重写
- 策略引擎重构
- 多智能体编排重构

---

## 4. 设计原则

### 4.1 业务级优先

工具应返回“业务结果 + 治理信息”，不是只返回底层原始对象。

### 4.2 统一包装优先

所有进入智能体的结果应由同一序列化/包装规则处理。

### 4.3 尽量可追踪

调用者应能看到：

- 工具名
- 类别
- 数据来源
- 质量信息
- 是否降级

---

## 5. 标准 Schema

```json
{
  "ok": true,
  "tool": "get_market_overview",
  "category": "market",
  "data_source": "mock",
  "summary": "已返回市场概览。",
  "data": {},
  "meta": {
    "payload_version": "1.0",
    "generated_at": "2026-06-17T00:00:00Z",
    "tool": "get_market_overview",
    "category": "market",
    "data_source": "mock",
    "governance": {}
  }
}
```

---

## 6. 验收标准

- 所有工具都能返回统一 schema
- `data_source` 在顶层可见
- `meta.governance` 可见
- 智能体不再需要区分不同工具的返回风格
- 缓存、降级、配置来源都能稳定表达

---

## 7. 归档要求

当以下内容完成后，可将本规格归档：

1. 工具层统一包装完成
2. 编排层可继续消费统一结果
3. 轻量验证通过
4. 形成归档文档
