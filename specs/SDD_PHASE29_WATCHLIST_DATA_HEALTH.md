## 1. 背景

Phase 28 已经把自选股日常决策清单做出来了，但实际跑起来时，真正影响使用体验的不是“有没有结果”，而是“结果靠不靠谱”。

项目当前已经具备数据治理层，也能标记 `real / mock / cache / config / unknown`，但这些信息还没有在自选股场景里形成一个开跑前的统一预检视图。

因此，本阶段不继续扩展分析能力，而是把数据健康状态收束成一个轻量、可直接读取的预检入口。

---

## 2. 目标

### 2.1 总目标

让自选股在进入日常决策之前，先经过一次数据健康预检，明确告诉用户哪些股票的数据是可直接使用的，哪些股票已经降级，哪些股票不适合作为当天的决策依据。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 可信度透明 | 明确显示历史行情、基本面数据的来源和降级状态 |
| 预检优先 | 先看数据健康，再看决策清单 |
| 低成本 | 复用现有治理层，不新增数据库或新算法 |
| 便于联调 | 一个脚本即可批量查看自选股数据健康 |
| 可导出 | 允许导出 JSON 供后续复盘或对比 |

## 4. 设计原则

### 4.1 先看健康，再看分析

如果数据本身已经降级，后续分析结果也要明确提示，而不是默默当成正常结果。

### 4.2 复用治理层

优先使用 `fetch_stock_hist_governed()` 和 `get_financial_indicators_governed()`，避免重复实现来源和质量判断。

### 4.3 输出分级

结果按“完全可用 / 部分降级 / 明显降级”分组，降低用户扫视成本。

### 4.4 保持轻量

只做预检，不做新的数据修复、不做数据仓库、不做复杂告警。

---

## 5. 输出约定

### 5.1 每只股票建议输出

- `stock_code`
- `name`
- `status`
- `history_source`
- `fundamental_source`
- `history_quality`
- `fundamental_quality`
- `history_reason`
- `fundamental_reason`
- `health_score`
- `summary`

### 5.2 状态定义

| 状态 | 含义 |
|------|------|
| 完全可用 | 历史行情和基本面均为可用真实数据 |
| 部分降级 | 至少有一条数据链路降级，但仍可参考 |
| 明显降级 | 关键链路降级，当前不建议直接依赖 |

---

## 6. 验收标准

- 可以批量读取 watchlist
- 可以看到每只股票的数据健康状态
- 可以区分历史行情和基本面链路
- 可以导出 JSON
- 不影响 Phase 28 的日常决策清单脚本
