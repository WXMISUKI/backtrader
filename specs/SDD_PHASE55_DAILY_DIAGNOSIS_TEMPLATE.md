## 1. 背景

Phase 53 和 Phase 54 已经把诊断口径做出来并贯通到主入口了。

但当前 `daily_summary` 的构建逻辑仍然集中在 `daily_watchlist_pipeline.py`，如果未来再有别的入口也要复用同样的摘要语义，就还会继续复制同一段逻辑。

因此，本阶段要做的不是增加新能力，而是把日常诊断摘要收成一个真正的公共模板，让多个入口只消费同一份摘要结构。

---

## 2. 目标

### 2.1 总目标

把日常诊断摘要抽成公共模板，减少重复拼接逻辑。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 模板统一 | 日常摘要只在一个公共位置构建 |
| 接口更薄 | 入口层只负责传入分组数据 |
| 语义一致 | `status / diagnosis_counts / summary_lines` 保持一致 |
| 便于扩展 | 后续其他入口可直接复用同一模板 |

---

---

## 4. 设计原则

### 4.1 只收口，不扩语义

模板只是把已有语义收拢，不重新定义新的业务规则。

### 4.2 入口变薄

调用方只负责把健康分组和决策分组传进去，不再重复拼摘要。

### 4.3 模板即接口

模板输出本身就是主入口要依赖的接口，稳定性优先。

---

## 5. 输出约定

### 5.1 日常诊断摘要模板输出

- `status`
- `health_counts`
- `decision_counts`
- `diagnosis_counts`
- `highlights`
- `summary_lines`
- `summary_text`

### 5.2 调用方式

调用方输入：

- `health_groups`
- `decision_groups`

调用方不再单独计算上述摘要字段。

---

## 6. 验收标准

- `build_daily_diagnosis_summary()` 可以独立产出完整摘要
- `daily_watchlist_pipeline.py` 不再重复实现同样的摘要拼接
- 统一工作流和验收入口仍然可正常运行
- README 能说明这是一份公共摘要模板
