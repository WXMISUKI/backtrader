## 1. 背景

Phase 63 已经把日常结果收成 `action_list`，Phase 64 又把行动提示语句说得更直接。

但在回看诊断时，用户仍然需要把“这几个样本为什么会落在这里”自己再拼一遍。

Phase 65 的目标就是把样本归因补齐，让诊断证据不只是有样本和原因，而是能直接回答：

> 这些样本主要是因为什么被归到这里？各自有多少？能不能顺手看出代表样本？

---

## 2. 总目标

在不增加复杂模型的前提下，把日常诊断证据里的样本整理成可直接消费的归因块，方便验收、回看和留档快速定位主要问题来源。

---

## 3. 核心原则

### 3.1 只做轻量归因

只做基于现有 `diagnosis`、`status`、`reason_codes` 和数据健康状态的归因，不引入复杂模型。

### 3.2 归因要可回看

归因结果必须能落到具体股票样本，方便用户直接验证。

### 3.3 归因要和现有摘要对齐

样本归因不能和 `diagnosis_counts`、`top_causes`、`sample_items` 语义冲突，而是对它们做更好读的整理。

---

## 4. 范围

### 4.1 纳入范围

- 新增 `sample_attribution`
- 把样本按主要原因分组
- 为每组补充代表样本
- 在留档查看、诊断收口包和投产验收中展示归因
- 更新运行说明、任务清单和归档

### 4.2 不纳入范围

- 新归因模型
- 收益归因
- 策略重训
- 自动交易

---

## 5. 归因定义

建议把归因整理成“原因 -> 样本组 -> 代表样本”的结构。

| 字段 | 含义 |
|---|---|
| `cause` | 主要归因原因 |
| `label` | 归因标签 |
| `count` | 归入该原因的样本数 |
| `examples` | 代表样本，便于快速核对 |
| `summary` | 面向用户的一句话总结 |

---

## 6. 规则约定

### 6.1 主要归因来自现有诊断

优先使用 `diagnosis.primary_cause`，必要时结合 `status`、`normalized_reasons` 和 `confidence_level` 做辅助说明。

### 6.2 代表样本要少而准

每个原因只保留少量代表样本，避免归因块膨胀。

### 6.3 归因块要能直接展示

终端、JSON 和留档查看都应能直接展示 `sample_attribution`。

---

## 7. 输出契约

建议在 `diagnosis_evidence` 中新增：

```json
{
  "diagnosis_evidence": {
    "sample_attribution": [
      {
        "cause": "history_mock_fallback",
        "label": "历史行情回退到 mock",
        "count": 5,
        "summary": "历史行情回退导致 5 个样本进入降级。",
        "examples": [
          {
            "stock_code": "000001",
            "name": "平安银行",
            "status": "部分降级",
            "primary_label": "历史行情回退到 mock"
          }
        ]
      }
    ]
  }
}
```

## 9. 验收标准

- `diagnosis_evidence` 中包含 `sample_attribution`
- 留档查看能展示主要样本归因
- 诊断收口包能展示主要样本归因
- 验收能识别样本归因存在

---

## 10. 后续建议

Phase 65 完成后，再往下优先：

1. 更稳定的日常节奏
2. 更细的提示语境联动

继续收的时候，尽量保持归因、提示、门禁三层语义各自清楚。
