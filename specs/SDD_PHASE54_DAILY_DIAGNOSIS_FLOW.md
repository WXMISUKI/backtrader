## 1. 背景

Phase 53 已经把健康摘要的降级原因分层出来了，但这些诊断信息还没有在所有日常主入口里完全一致地出现。

当前最有价值的收口，不是继续增加新的分析输出，而是把同一套诊断口径贯通到：

- 默认日常工作流
- 投产验收入口
- 日常统一流程的报告摘要

这样用户在不同入口看到的不是三种说法，而是同一套诊断答案。

---

## 2. 目标

### 2.1 总目标

让日常工作流、验收和统一流程都共享同一套诊断字段与诊断汇总。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 贯通一致 | 默认工作流、验收入口、统一流程看到同一套诊断口径 |
| 结果可扫 | 日常总览和验收摘要能直接看到主要诊断原因 |
| JSON 可用 | 诊断信息可继续被后续脚本消费 |
| 保持轻量 | 不新增分析模型，不新增数据源 |

---

## 3. 范围

### 3.1 纳入范围

- 默认工作流 JSON 增加诊断载荷
- 投产验收检查诊断摘要
- 日常统一流程总览补充诊断行
- 更新运行说明
- 更新任务清单
- 编写归档文档

### 3.2 不纳入范围

- 新增分析模型
- 新增数据源
- 新增前端页面
- 新增调度服务
- 新增自动修复逻辑

---

## 4. 设计原则

### 4.1 统一口径优先

不同入口都应该引用同一个诊断字段，而不是各自再做一套判断。

### 4.2 用于排查，不用于装饰

诊断信息应该直接帮助定位问题，不要只是让 JSON 更大。

### 4.3 保持薄层

只做传播、展示和校验，不重新计算业务结论。

---

## 5. 输出约定

### 5.1 默认工作流建议输出

- `daily_run_status_payload`
- `pipeline_payload`
- `diagnosis_counts`

### 5.2 投产验收建议输出

- `diagnosis_summary`
- `diagnosis_counts`
- `latest_payload`

### 5.3 日常统一流程建议输出

- `diagnosis_counts`
- `降级原因` 汇总行

---

## 6. 验收标准

- 默认工作流 JSON 能拿到诊断信息
- 投产验收能判断诊断摘要是否存在
- 日常统一流程总览能直接显示降级原因统计
- 原有流程仍然可独立运行
