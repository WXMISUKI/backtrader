# 软件设计规格说明 (SDD)
# Phase 90: 对话式入口产品化与最小聊天壳

---

## 1. 背景

当前项目已经具备较完整的量化能力、日常使用测试链路、回看与验收闭环，也具备统一编排层和智能体调用能力。

但从真实使用习惯看，用户更容易接受的是“像聊天一样用系统”：

- 直接输入自然语言
- 由智能体自己决定调用哪些工具
- 返回结构化结论、依据、风险和下一步动作
- 不要求用户先理解一堆脚本入口

目前项目虽然已经有 HTTP API、CLI、agent demo 和 orchestrator，但它们仍然更像“能力入口”，还没有收成一个真正顺手的对话产品。

Phase 90 的目标不是继续加分析能力，而是把现有能力收成一个最小可投产的对话式入口。

---

## 2. 总目标

把项目从“能力已具备”推进到“用户可以像聊天一样稳定使用”的状态。

要求达到：

- 用户能通过最小 CLI 入口持续对话
- 用户能通过统一 API 入口接入对话能力
- 对话结果能稳定展示工具调用、结论、依据、风险、下一步动作
- 后续 WebUI 可以直接复用同一套协议

---

## 3. 设计原则

### 3.1 先统一协议，再做外壳

不要先堆前端页面，再倒逼后端补协议。

### 3.2 先 CLI，再 WebUI

CLI 是最短投产路径，也是最适合验证交互协议的入口。

### 3.3 复用现有能力，不重写编排

优先复用：

- `core/agent/client.py`
- `core/orchestrator.py`
- `core/api_server.py`
- `examples/api_demo.py`
- `examples/agent_demo.py`

### 3.4 不做大而全聊天平台

本阶段只解决“好用的对话入口”，不引入复杂前端框架、流式编排平台或多租户系统。

---

## 4. 范围

### 4.1 纳入范围

- 统一对话请求 / 响应协议
- 最小 CLI 对话壳
- HTTP 对话入口语义收口
- 轻量化命令与会话体验
- 最小使用说明与归档

### 4.2 不纳入范围

- 完整 WebUI 产品
- 多会话管理后台
- 会话权限系统
- 复杂消息流式渲染
- 新一代编排层重构

---

## 5. 目标输出

Phase 90 重点输出以下能力：

- `chat_request_contract`
- `chat_response_contract`
- `cli_chat_shell`
- `api_chat_entry`
- `conversation_command_set`
- `tool_trace_summary`

---

## 6. 规则约定

### 6.1 对话输入

支持自然语言输入，不要求用户先选择分析模块。

### 6.2 对话输出

默认输出必须至少包含：

- 结论
- 依据
- 风险
- 下一步动作

如果存在工具调用轨迹，应额外展示：

- 调用了什么工具
- 数据源是什么
- 是否发生降级

### 6.3 会话命令

最小 CLI 应支持：

- `/help`
- `/capabilities`
- `/health`
- `/reset`
- `/exit`

### 6.4 失败语义

如果数据不足、工具失败或真实数据源不可用，必须明确说明，不允许包装成“已可用”。

## 8. 验收标准

- 用户可以通过 CLI 连续进行自然语言对话
- 用户可以用一句话触发分析、推荐、回看或能力查询
- 输出结构稳定，且明确区分事实、推理和建议
- 现有 API 能作为后续 WebUI 的统一后端
- 不破坏已有日常使用测试链路
