## 1. 背景

Phase 78 到 Phase 81 已经把日常投产基线收成了一个很薄的入口，并且具备：

- `failure_class`
- `repair_hint`
- `evidence_summary`
- `baseline_observation`

现在还缺最后一层最实用的东西：

> 最近这些问题，先修什么，怎么修，修完怎么验。

因此 Phase 82 只做一件事：把观测到的高频故障收成修复优先级，让基线入口能直接给出最小可执行的修复顺序。

---

## 2. 总目标

建立一个最小的修复优先级层，让系统每天都能快速回答：

- 先修哪一类问题
- 为什么要先修它
- 修完后怎么验证
- 哪些问题只是顺带处理，不能抢主修复位

本阶段只做修复排序和执行提示，不自动修复，不改主链路，不新增模型。

---

## 3. 设计原则

### 3.1 只收敛到可执行

输出必须能直接变成下一步动作，不只是“问题列表”。

### 3.2 优先处理阻断项

阻断门禁、执行产物缺失、provider 可见性缺失应优先于非阻断摘要问题。

### 3.3 保持薄层

只在基线入口增加 `repair_priority`，不重新设计工作流。

### 3.4 不制造自动化幻觉

本阶段只给建议和验证方式，不自动改文件、不自动重跑流程。

---

## 4. 范围

### 4.1 纳入范围

- 高频失败类别排序
- 修复目标排序
- 修复原因说明
- 验证建议
- 路线图与任务清单更新

### 4.2 不纳入范围

- 自动修复
- 新数据源
- 新模型
- 新前端
- 新调度系统

---

## 5. 建议输出

建议在基线 JSON 中新增 `repair_priority`，至少包含：

| 字段 | 含义 |
|---|---|
| `summary_text` | 一句话修复优先级结论 |
| `items` | 修复条目列表 |
| `top_target` | 当前第一优先修复项 |
| `history_path` | 参考的历史留档路径 |

每个修复条目至少包含：

| 字段 | 含义 |
|---|---|
| `target` | 修复目标 |
| `failure_class` | 对应失败类别 |
| `priority` | 优先级数字，越小越优先 |
| `reason` | 为什么先修它 |
| `command_hint` | 可能的操作提示 |
| `verify_hint` | 修完怎么验证 |

---

## 6. 规则约定

### 6.1 阻断优先

`gate_missing`、`execution_missing`、`provider_visibility_missing` 优先级应高于 `partial_artifacts`。

### 6.2 趋势优先

最近 N 次中出现频率更高的失败类别，优先级应上调。

### 6.3 可验证

每个修复建议必须带验证提示，避免“修了但不知道是否修好”。

### 6.4 不越权

如果当前状态已经是 `ready`，则修复优先级应退化为保持观察，不应制造额外动作。

## 8. 验收标准

- 能看到当前第一优先修复项
- 能看到修复理由和验证方式
- 高频阻断项会被优先排序
- 不破坏 `ready / caution / failed / blocked` 语义

---

## 9. 后续建议

Phase 82 完成后，再观察修复优先级在真实日常中的稳定性，然后再考虑是否把它透传到提示词、回看入口或验收入口。
