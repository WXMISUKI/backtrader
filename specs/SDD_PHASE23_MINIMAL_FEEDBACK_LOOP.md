## 1. 背景

当前项目已经具备：

- 统一决策入口
- 标准工作流总入口
- 决策会话
- 用户反馈
- 回放与统计

但对于“边用边改”的私用场景来说，反馈入口还可以更薄一些：

- 用户不想每次都填很多字段
- 最常见的反馈其实只有“采纳 / 不采纳”
- 有时候只知道 `session_id`，不想再额外找 `workflow_id`

因此，本阶段要把反馈闭环做成一个最小、直接、好用的入口。

---

## 2. 目标

### 2.1 总目标

提供一个最小反馈入口，让用户可以快速记录某次决策的采纳情况，并保持与会话、工作流和统计链路的关联。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 最小输入 | 允许仅传 `session_id` 即可提交反馈 |
| 低门槛 | 默认支持采纳 / 不采纳 / 评分 / 备注 |
| 可追踪 | 反馈仍能关联 `workflow_id` 和会话记录 |
| 可复盘 | 反馈能进入回放与统计 |
| 快速迭代 | 不引入复杂反馈模型 |

---

## 3. 范围

### 3.1 纳入范围

- `POST /feedback`
- `submit_decision_feedback`
- `DecisionSessionStore.submit_feedback`
- `StockOrchestrator.submit_decision_feedback`
- `StockAgentRuntime.submit_decision_feedback`

### 3.2 不纳入范围

- 多轮反馈协议
- 复杂评分模型
- 独立反馈工作台
- 新权限系统

---

## 4. 设计原则

### 4.1 最小字段

反馈优先保留最少必要字段，不让联调变复杂。

### 4.2 兼容现有链路

如果已有 `workflow_id` 就直接使用；如果没有，则尝试从 `session_id` 回推。

### 4.3 采纳优先

最常见使用场景是记录“采纳 / 不采纳”，评分和备注是补充，不是前置条件。

### 4.4 只做薄层

反馈入口只负责写入和透传，不新增复杂判断逻辑。

---

## 5. 接口约定

### 5.1 `POST /feedback`

请求体建议：

- `session_id`
- `workflow_id`（可选）
- `accepted`
- `rating`
- `reason`
- `correction`
- `comment`

其中 `session_id` 必填，`workflow_id` 可选。

### 5.2 反馈回退规则

如果请求里没有 `workflow_id`，系统应优先尝试从 `session_id` 关联的会话中回推 `workflow_id`。

### 5.3 输出要求

反馈结果至少包含：

- `feedback_id`
- `session_id`
- `workflow_id`
- `accepted`
- `rating`
- `reason`
- `correction`
- `comment`

---

## 6. 验收标准

- `POST /feedback` 允许只传 `session_id`
- 已有会话时可以自动补齐 `workflow_id`
- 反馈可以正常写入 JSONL
- 反馈可以进入会话回放和统计
- 不破坏现有 `/decision` 流程
