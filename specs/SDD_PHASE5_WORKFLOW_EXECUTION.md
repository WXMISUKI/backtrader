## 1. 背景

项目目前已经具备：

- 共享路由
- 复合意图优先级
- 路由审计
- 协作规划
- 统一工具输出

下一阶段最有价值的不是继续增加单点能力，而是把这些能力组织成可执行的工作流闭环。

如果没有工作流执行层，协作规划只能停留在“建议怎么做”，无法回答：

- 谁先执行
- 哪些结果可以直接复用
- 哪些步骤失败后可以继续
- 最终结果如何统一输出
- 这次请求到底经历了什么

因此需要一个轻量的工作流执行引擎。

---

## 2. 目标

### 2.1 总目标

让协作规划结果可以被执行成一条可追踪、可回放、可审计的业务链路。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 可执行 | 协作计划能转成实际工具调用 |
| 可串联 | 多个工具能按顺序或依赖执行 |
| 可回放 | 每一步的输入输出可审计 |
| 可降级 | 支持任务失败时保留主结果 |
| 可统一 | 最终结果符合统一输出信封 |

---

## 3. 范围

### 3.1 纳入范围

- 协作计划到执行计划的转换
- 工作流节点执行
- 节点结果聚合
- 工作流审计
- 工作流统一输出
- 复合意图自动触发工作流执行

### 3.2 不纳入范围

- 分布式任务编排
- 消息队列式异步调度
- 跨会话持久化依赖图
- 人工审批流

---

## 4. 设计原则

### 4.1 先计划，再执行

协作计划是执行的输入，不是最终结果。

### 4.2 节点可替换

每个工作流节点都应该对应已有业务工具，避免重复写业务逻辑。

### 4.3 失败可降级

支持任务失败不应破坏主任务的结果输出。

### 4.4 审计优先

工作流的执行顺序、结果和失败原因必须可回放。

---

## 5. 核心设计

### 5.1 工作流执行器

建议新增 `core/agent/workflow.py`，负责：

- 接收协作计划
- 按顺序执行任务
- 聚合节点结果
- 生成工作流结果

### 5.2 工作流入口

建议暴露以下入口：

```python
execute_workflow(user_input: str, risk_profile: str = "moderate") -> dict
```

### 5.3 适合自动工作流化的场景

| 场景 | 说明 |
|------|------|
| 市场概览 + 风控 + 分析 | 先补背景，再给结论 |
| 推荐 + 回测 + 报告 | 先推荐，再验证，再输出 |
| 选股 + 基本面 + 风控 | 先筛候选，再做判断 |

### 5.4 工作流节点字段

建议每个节点至少具备：

- `id`
- `intent`
- `tool`
- `role`
- `description`
- `arguments`
- `depends_on`
- `status`
- `summary`
- `data_source`
- `error`

### 5.5 工作流结果字段

建议最终结果至少具备：

- `workflow_id`
- `objective`
- `mode`
- `plan`
- `steps`
- `execution_order`
- `summary`
- `is_degraded`
- `route_audit_id`
- `meta`

### 6.1 编排器接口

```python
        pass
```

```python
class ArkAgentClient:
    def execute_workflow(self, user_input: str) -> dict:
        pass
```

### 6.3 工具接口

建议增加智能体工具：

- `execute_workflow`

---

## 7. 验收标准

- 复合意图可以直接进入工作流执行
- 工作流输出包含计划和执行结果
- 工作流执行有审计记录
- 支持任务失败时主结果可用
- 现有单工具链路不被破坏
