## 1. 背景

Phase 40 到 Phase 51 已经把自选股日常链路收得比较完整了：

- `daily_run` 负责运行和归档
- `review` 负责看最新留档和反馈趋势
- `acceptance` 负责投产验收

但这些入口仍然需要用户自己按顺序记住并执行。

对私人私用、快速投产的项目来说，最省心的状态不是“功能很多”，而是“有一个默认工作流命令串，直接跑就行”。

因此，本阶段不新增分析能力，也不改任何业务逻辑，而是把现有三段式流程收成一个默认日常工作流入口。

---

## 2. 目标

### 2.1 总目标

提供一个最短路径的默认日常工作流，把运行、回看和验收串成一个命令。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 一条命令 | 可以直接跑完 `daily_run + review + acceptance` |
| 默认顺序 | 先运行，再回看，最后验收 |
| 保持薄层 | 只编排，不重算，不重写结果 |
| 可单独回退 | 原有三个入口仍然保留 |
| 适合每天用 | 用户不必再手工记多个命令 |

## 4. 设计原则

### 4.1 只编排，不发明

工作流入口只负责调度现有脚本，不重新实现任何业务步骤。

### 4.2 默认顺序清晰

默认顺序应该是：

1. `daily_run`
2. `review`
3. `acceptance`

### 4.3 保持轻量

输出应清楚说明每一步是否成功，但不要变成复杂日志面板。

### 4.4 失败可见

任一步失败都应在终端中明确表现出来，便于快速排查。

---

## 5. 输入输出约定

### 5.1 输入

建议支持：

- `--watchlist`
- `--portfolio`
- `--archive-dir`
- `--output-json`
- `--skip-review`
- `--skip-acceptance`
- `--show-json`

### 5.2 输出

工作流入口至少应输出：

- `status`
- `daily_run_code`
- `review_code`
- `acceptance_code`
- `summary`
- `output_json`

---

## 6. 验收标准

- 可以一条命令跑完日常运行、回看和验收
- 任一步失败时都有清晰退出码
- 原有三个入口仍可独立使用
- README 能明确说明默认工作流命令
