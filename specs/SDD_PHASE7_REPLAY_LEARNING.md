## 1. 背景

Phase 6 已经建立统一任务协议，使协作计划、工作流步骤、工作流结果开始共享同一套结构语义。

但如果只有统一协议，还不足以形成长期演进能力。项目还需要回答两个问题：

1. 过去某次任务到底是怎么跑的
2. 这些任务模式能不能被统计、复盘和学习

当前已有的底座包括：

- 路由审计 JSONL
- 模板命中统计 JSONL
- 运行监控 JSONL
- 工作流 ID 与步骤级审计

因此 Phase 7 不追求重型分析平台，而是补齐最小可用的回放与学习统计闭环。

---

## 2. 目标

### 2.1 总目标

让智能体与编排器可以按 `workflow_id` 回放任务链路，并按任务模式做轻量统计。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 可回放 | 能按 `workflow_id` 看到规划、步骤、结果和模板命中 |
| 可统计 | 能统计任务回放数量、模板命中情况、失败/降级比例 |
| 可复用 | 回放和统计结果都使用统一任务协议字段 |
| 可解释 | 能看到任务为什么成功、失败或降级 |
| 可轻量 | 继续使用 JSONL 和内存聚合，不引入重型存储 |

---

## 3. 范围

### 3.1 纳入范围

- 任务回放查询
- 任务链路汇总
- workflow 级统计
- 模板命中统计
- 降级和失败统计
- 可消费的统一输出

### 3.2 不纳入范围

- 模型训练
- 向量检索
- 自动标签学习
- 外部数据库接入

---

## 4. 核心对象

### 4.1 ReplayTrace

一次回放结果，用于展示完整任务链路。

建议字段：

- `workflow_id`
- `task_id`
- `objective`
- `plan`
- `steps`
- `result`
- `template`
- `audit`
- `summary`
- `meta`

### 4.2 LearningStats

轻量学习统计结果，用于看趋势和命中情况。

建议字段：

- `total_workflows`
- `total_steps`
- `template_hit_count`
- `degraded_count`
- `failed_count`
- `intent_counts`
- `tool_counts`
- `template_counts`
- `recent_workflows`

## 6. 功能设计

### 6.1 任务回放

应支持：

- 按 `workflow_id` 查询整条链路
- 按事件类型过滤
- 按阶段过滤
- 返回最近 N 条回放摘要

### 6.2 任务统计

应支持：

- 回放总数
- 模板命中率
- 降级率
- 失败率
- 常见意图
- 常见工具
- 最近回放样本

### 6.3 任务学习

学习层只做轻量统计，不做模型训练：

- 记录出现频次
- 记录模板命中表现
- 记录降级和失败特征
- 为后续路由和模板调整提供依据

---

## 7. 输出要求

### 7.1 回放输出

回放结果应包含：

- 原始审计片段
- 统一任务协议视图
- 任务摘要
- 降级标记

### 7.2 统计输出

统计结果应包含：

- 汇总数据
- 最近样本
- 分类计数
- 可继续消费的 `data_source`
