## 1. 背景

项目目前已经具备：

- 共享路由
- 协作规划
- 工作流执行
- 模板工作流
- 模型治理

但如果没有运行保障层，我们仍然只能知道“功能做出来了”，不能及时知道：

- 哪些入口成功率下降了
- 哪些链路开始频繁降级
- 哪些工具响应变慢了
- 哪些异常需要主动告警

因此需要一层轻量的运行保障和监控告警能力，作为整个 Phase 5 的稳定性底座。

---

## 2. 目标

### 2.1 总目标

建立一套轻量、可查询、可触发告警的运行保障能力，用于监控数据、分析、模型和系统链路的关键健康指标。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 指标收集 | 记录关键业务和系统指标 |
| 告警触发 | 基于阈值自动生成告警 |
| 状态查询 | 可以查看当前健康状态和最近事件 |
| 运行追踪 | 可以回看最近的异常与告警 |
| 轻量落地 | 本地持久化，不引入重型监控平台 |

---

## 3. 范围

### 3.1 纳入范围

- 指标收集
- 告警规则
- 告警记录
- 运行健康查询
- 路由 / 工作流 / 编排链路的关键运行事件

### 3.2 不纳入范围

- Prometheus / Grafana / OpenTelemetry 全套接入
- 分布式 tracing
- 复杂告警编排和多级路由

---

## 4. 设计原则

### 4.1 先收口，再扩展

先统一收集关键指标和告警，再考虑更复杂的外部平台接入。

### 4.2 轻量优先

优先采用本地内存聚合 + JSONL 持久化，避免额外基础设施依赖。

### 4.3 与业务解耦

监控层只负责记录和判断，不替代业务决策逻辑。

### 4.4 可解释

每条告警都应该说明触发原因和对应指标。

---

## 5. 核心设计

### 5.1 指标模型

建议至少支持以下指标口径：

- 请求总数
- 成功数
- 失败数
- 降级数
- 缓存命中数
- 平均延迟
- 最大延迟
- 错误率
- 降级率
- 缓存命中率

### 5.2 告警模型

建议每条告警包含：

- `alert_id`
- `name`
- `severity`
- `status`
- `reason`
- `metric_name`
- `metric_value`
- `threshold`
- `triggered_at`

### 5.3 运行健康

健康状态至少分为：

- `ok`
- `degraded`
- `warning`
- `critical`

健康状态应综合指标和最近告警判断。

---

## 6. 对外接口

### 6.1 监控服务

建议新增：

```python
class ObservabilityService:
    def record_metric(...)
    def record_latency(...)
    def record_runtime_event(...)
    def evaluate(...)
    def get_health(...)
    def recent_alerts(...)
```

### 6.2 智能体工具

建议新增：

- `get_runtime_health`
- `evaluate_runtime_health`

### 6.3 编排层接口

建议编排器和工作流执行器在关键完成点上回写运行指标。

---

## 7. 监控对象

优先监控以下链路：

- `route()`
- `plan_collaboration()`
- `execute_workflow()`
- 推荐/分析/回测工具调用
- 模型治理评估

---

## 8. 验收标准

- 关键链路可以写入运行指标
- 可基于阈值生成告警
- 可查询当前健康状态
- 可查看最近告警
- 现有主业务链路不被破坏

---

## 9. 归档要求

当以下内容完成后，可将本规格归档：

1. 运行保障服务实现完成
2. 智能体监控工具接入完成
3. 编排或工作流链路至少一个入口写入指标
4. 轻量验证通过
5. 形成归档文档
