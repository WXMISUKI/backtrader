## 1. 背景

当前项目已经具备：

- 统一编排
- 数据治理
- 协作规划
- 工作流执行
- 标准工作流模板

但模型相关能力仍以“直接调用算法”为主，缺少对以下问题的统一治理：

- 当前用的是哪个模型版本
- 模型训练时用了哪些特征
- 模型结果是否达标
- 哪个版本可以发布
- 出现问题时如何回滚

如果没有这一层，项目只能说明“模型能跑”，不能说明“模型可信、可发布、可追溯”。

---

## 2. 目标

### 2.1 总目标

围绕现有推荐与在线学习能力，建立一层最小可用的模型治理闭环，让模型从“可用”走向“可管理”。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 版本可追溯 | 每次保存的模型有明确版本和时间戳 |
| 特征可追踪 | 能记录训练时的特征集合和顺序 |
| 评估可判断 | 能基于阈值判断模型是否达标 |
| 发布可控制 | 未达标模型不得标记为稳定版本 |
| 回滚可执行 | 出现问题时能回到上一个稳定版本 |

---

## 3. 范围

### 3.1 纳入范围

- 模型版本注册
- 特征清单管理
- 模型评估与门禁
- 模型发布状态
- 模型回滚
- 模型治理查询入口

### 3.2 不纳入范围

- 复杂 MLOps 平台
- 分布式模型仓库
- 自动超参搜索
- 全量训练流水线编排

---

## 4. 设计原则

### 4.1 先治理，再扩展

先把模型版本、特征、门禁和回滚做实，再考虑更复杂的训练编排。

### 4.2 轻量落地

治理信息采用本地 JSON 持久化，优先保证可用和易调试。

### 4.3 与算法解耦

治理层不改模型算法本身，只包裹元数据、阈值和发布状态。

### 4.4 可回放

所有治理决策都应能通过查询接口和审计记录复原。

---

## 5. 核心设计

### 5.1 模型注册

每个模型记录应至少包含：

- `model_name`
- `version`
- `artifact_path`
- `status`
- `metrics`
- `thresholds`
- `feature_manifest`
- `created_at`

### 5.2 特征清单

每个特征清单应至少包含：

- `manifest_id`
- `name`
- `version`
- `features`
- `feature_hash`
- `description`

### 5.3 评估门禁

模型发布前应进行阈值判断：

- 命中阈值则可标记为 `stable`
- 未命中阈值则保持 `draft` 或 `blocked`
- 所有失败原因必须可见

### 5.4 回滚策略

当稳定版本异常时，应能回退到同一模型族下的上一个稳定版本。

---

## 6. 对外接口

### 6.1 治理服务

建议新增：

```python
class ModelGovernanceService:
    def register_feature_manifest(...)
    def register_model(...)
    def evaluate_release(...)
    def promote(...)
    def rollback(...)
    def get_status(...)
```

### 6.2 智能体工具

建议新增：

- `get_model_governance_status`
- `evaluate_model_release`

### 6.3 编排层接口

建议编排器可直接查看模型治理状态，便于后续把发布门禁与分析结果联动。

---

## 7. 适用对象

当前优先治理的对象为：

- `skills/stock_recommender/ml_recommender.py`
- `skills/stock_recommender/online_learner.py`

---

## 8. 验收标准

- 模型保存后能注册治理元数据
- 特征清单可查询
- 发布前评估可执行
- 未达标模型不能被标记为稳定
- 可以查询模型治理状态
- 现有推荐逻辑不被破坏

---

## 9. 归档要求

当以下内容完成后，可将本规格归档：

1. 模型治理服务实现完成
2. 推荐器或在线学习器接入治理元数据
3. 模型治理查询工具接入完成
4. 轻量验证通过
5. 形成归档文档
