## 1. 背景

Phase 60 已经提供了统一投产门禁，Phase 61 已经增强了数据可信度，Phase 62 已经把反馈效果纳入回看。

但日常使用时，最有价值的并不是这些中间结果本身，而是：

> 今天到底该做什么。

Phase 63 的目标，就是把 `重点关注 / 继续观察 / 暂不行动 / 数据不足` 和投产门禁，进一步收成可执行的行动清单：

- `review_now`
- `hold_watch`
- `wait`
- `skip_due_to_data`
- `diagnose`

---

## 2. 总目标

建立一个最小可用的日常行动清单层，让系统在每次运行后能直接输出：

- 最优先处理的股票
- 需要继续持有观察的股票
- 可以先等待的股票
- 因数据不足而跳过的股票
- 在门禁阻断时应优先诊断的项目

---

## 3. 核心原则

### 3.1 行动优先

清单的目标不是再描述结果，而是把结果压成行动。

### 3.2 先门禁，后行动

所有行动都必须先读取 `production_gate`，门禁为 `block` 时不能输出强行动建议。

### 3.3 保持可读

终端必须能直接读懂前几条优先行动，JSON 则保留完整证据。

---

## 4. 范围

### 4.1 纳入范围

- 新增 `action_list` 构建能力
- 把行动清单接入日常统一流程
- 把行动清单接入诊断收口包
- 把行动清单接入留档查看入口

### 4.2 不纳入范围

- 新策略
- 自动交易
- 新前端
- 复杂排序模型

---

## 5. 行动定义

| 行动 | 含义 |
|---|---|
| `review_now` | 今天优先复核，重点看这只票 |
| `hold_watch` | 已持有或需要保持观察，关注仓位和风险 |
| `wait` | 暂时等待，不做强动作 |
| `skip_due_to_data` | 由于数据不足或可信度低，跳过本轮行动 |
| `diagnose` | 门禁阻断或数据异常时，优先排查问题 |

---

## 6. 规则约定

### 6.1 重点关注

- `重点关注` 且数据可信度足够时，进入 `review_now`

### 6.2 已持有标的

- `继续观察` 且处于持仓中时，进入 `hold_watch`

### 6.3 非持有观察

- `继续观察` 且未持有时，进入 `wait`

### 6.4 数据不足

- `数据不足` 或可信度过低时，进入 `skip_due_to_data`

### 6.5 门禁阻断

- `production_gate = block` 时，优先 `diagnose`
- 强行动建议必须被压掉

---

## 7. 输出契约

建议新增：

```json
{
  "action_list": {
    "status": "block",
    "counts": {
      "review_now": 1,
      "hold_watch": 2,
      "wait": 1,
      "skip_due_to_data": 1,
      "diagnose": 0
    },
    "items": [],
    "groups": {},
    "summary_text": "行动分布: ..."
  }
}
```

## 9. 验收标准

- 统一流程 JSON 中包含 `action_list`
- 默认流程和回看入口都能展示行动清单
- 规则能把重点关注、持仓观察、等待、跳过和诊断区分开
- 门禁为 `block` 时不会输出强行动建议

---

## 10. 后续建议

Phase 63 完成后，日常主线已经基本收口。

后续如果继续增强，优先从：

1. 更细的行动提示
2. 更清晰的样本归因
3. 更稳定的日常节奏

这三个方向里继续收，而不是继续堆入口。
