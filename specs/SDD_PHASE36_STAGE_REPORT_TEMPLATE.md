## 1. 背景

Phase 31 到 Phase 35 已经把自选股日常流程、日更报告、日报对比和周报模板都串起来了。

这意味着系统已经具备了持续留档和阶段回看的基础。但对于更长周期的复盘，比如按月、按半月、按一次策略迭代周期来看，周报还不够高一层。

阶段报告的价值在于：

- 总结一个较完整周期内的变化
- 归纳主要关注标的的演进
- 看出持仓语境和数据健康的长期状态
- 为下一阶段开发提供稳定复盘依据

因此，本阶段不再做周视图，而是把多份周报或多份日报再收成阶段报告模板。

---

## 2. 目标

### 2.1 总目标

把连续的周报或日报整理成一份可复用的阶段报告模板，适合按月或按项目阶段留档。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 阶段视图 | 让用户看到一段周期内的总体变化 |
| 复盘友好 | 报告适合留档、分享和回看 |
| 轻量聚合 | 只聚合已有周报/日报，不重算分析结果 |
| 保持兼容 | 不破坏现有日报和周报流程 |
| 模板稳定 | 便于后续重复使用 |

## 4. 设计原则

### 4.1 阶段报告来自周报/日报

阶段报告只聚合已有周报或日报，不定义新的分析口径。

### 4.2 输出结构固定

阶段报告段落顺序固定，便于长期复盘和横向比较。

### 4.3 复用已有结构

阶段报告基于 `weekly_report`、`daily_report`、`daily_comparison` 和累计结果生成。

### 4.4 不引入复杂状态

优先使用本地 JSON 列表作为输入，不引入数据库或后台服务。

---

## 5. 输入输出约定

### 5.1 输入

建议支持：

- `--stage-reports`
- `--stage-from`
- `--stage-to`

其中 `stage-reports` 可以是多个周报或日报 JSON 路径。

### 5.2 输出

建议新增 `stage_report` 对象，包含：

- `period`
- `headline`
- `summary`
- `health_summary`
- `decision_summary`
- `portfolio_summary`
- `trend_summary`
- `highlights`
- `report_text`

阶段报告文本建议包含：

- 阶段总览
- 关键变化
- 数据健康趋势
- 决策与持仓趋势
- 本阶段重点标的
- 下一阶段建议

---

## 6. 验收标准

- 可以把多份周报或日报聚合成阶段报告模板
- 阶段报告能反映一段周期内的总体变化
- 阶段报告文本适合留档
- JSON 中保留阶段报告结构
- 不影响现有日报和周报产物
