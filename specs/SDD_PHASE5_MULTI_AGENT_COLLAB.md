## 1. 背景

当前项目已经完成共享路由、复合意图优先级和路由审计。

这意味着系统已经能够回答两个关键问题：

- 用户想做什么
- 这次路由为什么选了这个工具

但我们还缺少第三个能力：

- 这次请求应该由哪些能力协同完成

对于更复杂的自然语言需求，单一工具虽然可用，但很容易出现：

- 只看见一个主意图，忽略上下文
- 把需要前置信息的任务直接执行掉
- 不能把市场、风控、分析、回测、报告串成完整链路

因此需要一个轻量的多智能体协作规划层。

---

## 2. 目标

### 2.1 总目标

为智能体接入补上一层“协作规划”能力，让系统在面对复合意图时，先生成结构化协作计划，再决定工具调用顺序。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 协作可解释 | 说明为什么要拆成多个任务 |
| 协作可规划 | 输出主任务、支持任务和执行顺序 |
| 协作可复用 | 路由、审计、工具调用复用现有能力 |
| 协作低侵入 | 不破坏现有 `ask()` / `route()` 接口 |
| 协作可回放 | 计划结果可被审计和查看 |

---

## 3. 范围

### 3.1 纳入范围

- 复合意图协作规划
- 主任务与支持任务拆解
- 角色建议与执行顺序
- 协作计划审计
- 智能体首轮优先进入规划器

### 3.2 不纳入范围

- 真正的分布式多智能体调度
- 子 agent 生命周期管理
- 远程任务队列或消息总线
- 跨会话记忆管理

---

## 4. 设计原则

### 4.1 先规划，再执行

复杂请求先转成结构化计划，再决定执行哪一组工具。

### 4.2 主路由不丢失

协作规划不能替代现有共享路由，只能在其基础上增加任务拆解。

### 4.3 支持任务可见

市场概览、风控检查、基本面分析等支持任务必须保留，不允许只输出最终结论。

### 4.4 计划优先可解释

规划结果优先给出：

- 主目标
- 主工具
- 支持角色
- 执行顺序
- 失败回退方案

## 5. 核心设计

### 5.1 新增协作规划模块

建议新增 `core/agent/collaboration.py`，用于根据自然语言和路由结果生成协作计划。

#### 计划输出建议字段

- `objective`
- `mode`
- `primary_intent`
- `primary_tool`
- `primary_reason`
- `roles`
- `tasks`
- `execution_order`
- `fallback`
- `candidates`
- `matched_terms`
- `meta`

### 5.2 协作任务建议

| 意图 | 建议角色 | 说明 |
|------|----------|------|
| `market_overview` | `market_analyst` | 负责市场背景和情绪 |
| `risk_profile` | `risk_guard` | 负责仓位、风险和约束 |
| `fundamental` | `fundamental_analyst` | 负责基本面补充 |
| `analyze_stock` | `equity_analyst` | 负责个股分析 |
| `recommend*` | `portfolio_advisor` | 负责推荐整理 |
| `report` | `report_writer` | 负责结果归档输出 |
| `backtest` | `backtest_executor` | 负责复验策略结果 |
| `screen_stocks` | `stock_screener` | 负责候选池筛选 |

### 5.3 协作触发条件

以下情况建议进入协作规划：

- 同一句话命中多个候选意图
- 同一句话包含明显的并列关系词
- 请求同时包含背景信息和结果输出
- 请求同时包含分析和回测
- 请求同时包含风控和报告

### 5.4 工具接入方式

建议新增一个对智能体暴露的工具：

- `plan_collaboration`

该工具只做规划，不直接执行分析结果，返回协作计划供后续工具调用。

## 6. 接口设计

### 6.1 协作规划入口

```python
build_collaboration_plan(user_input: str, default_risk_profile: str = "moderate") -> dict
```

### 6.2 编排层入口

```python
class StockOrchestrator:
    def plan_collaboration(self, user_input: str, **kwargs) -> dict:
        pass
```

### 6.3 运行时入口

```python
class StockAgentRuntime:
    def plan_collaboration(self, question: str) -> dict:
        pass

    def execute_workflow(self, question: str) -> dict:
        pass
```

### 6.4 智能体客户端行为

当路由结果显示为复合意图时，客户端首轮优先调用 `plan_collaboration`，由规划结果引导后续工具调用。

当规划已经足够明确时，可以进一步切换到 `execute_workflow`，直接把计划转成可执行工作流。

## 10. 与后续规格的关系

本规格负责“怎么拆”。

下一步的 `Phase 5 协作执行闭环与工作流引擎` 负责“怎么做”。

---

## 7. 统一输出要求

协作计划输出应保持与现有工具一致的信封结构：

- `ok`
- `tool`
- `category`
- `data_source`
- `summary`
- `data`
- `meta`

协作计划的 `data` 内部应包含：

- 主路由信息
- 支持任务信息
- 任务顺序
- 角色建议
- 回退方案

---

## 8. 验收标准

- 复合意图可以生成结构化协作计划
- 单一意图仍能直接走原有工具
- 客户端和编排器都可查看协作计划
- 规划结果可被审计
- 不影响既有路由、回测和报告链路
