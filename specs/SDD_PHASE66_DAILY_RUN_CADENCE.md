## 1. 背景

Phase 60 到 Phase 65 已经把“能不能用、今天先做什么、样本主要因为什么”收得比较完整。

但日常真正使用时，用户还会遇到一个更实际的问题：

> 今天这一轮运行到底完整不完整，接下来应该先看哪一步？

Phase 66 的目标不是再加解释字段，而是把每日运行节奏收成一条稳定的作业流，让默认入口更像固定日常，而不是一串脚本拼接。

---

## 2. 总目标

把 `daily_watchlist_daily_run.py` 收束成可读、可追踪、可复用的日常节奏入口，明确：

- 今天跑了哪几步
- 哪一步成功、哪一步降级、哪一步失败
- 现在用户应该先看什么

---

## 3. 核心原则

### 3.1 节奏先于扩展

先让日常入口稳定，再考虑更复杂的调度和自动化。

### 3.2 不自动修复

本阶段只描述运行节奏和状态，不引入自动修复或自动重试编排。

### 3.3 终端简短，JSON 完整

终端只显示关键节奏信息，JSON 保留完整状态和证据。

---

## 4. 范围

### 4.1 纳入范围

- 为 `daily_run` 增加统一节奏摘要
- 让运行状态明确展示 `preflight / pipeline / archive / view`
- 给出一个稳定的下一步建议
- 让默认工作流和验收入口能读取该节奏摘要
- 更新运行说明、任务清单和归档

### 4.2 不纳入范围

- 新调度平台
- 自动修复逻辑
- 复杂重试策略
- 前端工作台

---

## 5. 节奏定义

建议把日常运行拆成以下步骤：

| 步骤 | 含义 |
|---|---|
| `preflight` | 数据健康预检 |
| `pipeline` | 统一日常流程执行 |
| `archive` | 留档写入与 latest 更新 |
| `view` | 留档查看入口展示 |

---

## 6. 规则约定

### 6.1 节奏状态

每一步至少给出：

- `ok`
- `degraded`
- `failed`

### 6.2 下一步建议

运行结束后，应给出简短下一步建议，例如：

- 先看 `production_gate`
- 再看 `action_list`
- 如果失败，先看预检或 cookie

### 6.3 与现有状态对齐

`daily_run_status`、`production_gate`、`action_list`、`sample_attribution` 仍保留，节奏摘要只是把它们串起来。

---

## 7. 输出契约

建议在 `daily_watchlist_run_status.json` 中新增：

```json
{
  "run_cadence": {
    "steps": [
      {"name": "preflight", "status": "ok"},
      {"name": "pipeline", "status": "ok"},
      {"name": "archive", "status": "ok"},
      {"name": "view", "status": "ok"}
    ],
    "summary_text": "今日运行节奏完整：预检、执行、归档、查看都已完成。",
    "next_step": "先看 production_gate，再看 action_list。"
  }
}
```

- `run_status` 中包含 `run_cadence`

- 失败时能明确下一步先看什么
