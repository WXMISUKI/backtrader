## 1. 背景

Phase 33 已经把统一入口整理成稳定的日更报告，适合留档和回看。

但如果报告每天都只是一份独立快照，长期使用时仍然会有一个问题：看得到今天，但不容易立刻看出和前一天相比发生了什么变化。

这会削弱日报最重要的一个价值：

- 哪些标的进入/退出重点关注
- 健康状态是否改善或恶化
- 持仓语境是否发生变化
- 今日总体判断相比昨天是否更谨慎或更乐观

因此，本阶段的目标不是再做一份新报告，而是让现有日报支持对比，变成更适合长期复盘的日更记录。

---

## 2. 目标

### 2.1 总目标

让日更报告支持“今天 vs 上一次”的对比视图，帮助用户快速识别变化。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 识别变化 | 让用户一眼看到变化项 |
| 保持轻量 | 只做对比，不做时间序列分析 |
| 可留档 | 对比结果可写入 JSON |
| 兼容现有流程 | 不破坏当前日更报告输出 |
| 可继续扩展 | 后续可自然扩展到周对比或月对比 |

## 4. 设计原则

### 4.1 对比优先

核心输出是变化，而不是重复展示静态事实。

### 4.2 使用最近一次记录

先支持与上一次报告对比，控制复杂度。

### 4.3 复用现有结果

对比只基于现有 `daily_report`、`health`、`decision` 和 `portfolio_summary`。

### 4.4 不引入过重状态

采用本地 JSON 留档作为对比源，避免新增数据库或后台服务。

---

## 5. 输入输出约定

### 5.1 输入

继续使用 Phase 33 的统一入口输入，并新增可选参数：

- `--compare-with`

### 5.2 输出

建议新增 `comparison` 对象，包含：

- `baseline_path`
- `changed_sections`
- `summary_text`
- `delta_items`

对比重点建议包括：

- `daily_report.headline`
- `daily_summary.status`
- `health` 分布
- `decision` 分布
- `highlights`

---

## 6. 验收标准

- 能把当前报告与上一份报告对比
- 能输出变化项摘要
- 不影响当前日报和明细输出
- JSON 中保留对比对象
- 不引入重型存储依赖
