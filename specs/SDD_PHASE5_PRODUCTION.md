## 1. 引言

### 1.1 目的

本文档描述 Phase 5 的设计草案。该阶段不再以“单点功能扩展”为主，而是围绕现有分析、推荐、模拟、报告与可视化能力，构建一条可持续运行、可观测、可回归验证的生产化闭环。

### 1.2 范围

本阶段聚焦以下能力：
- 统一编排个股分析、推荐、回测、模拟交易与报告输出
- 强化真实数据链路的稳定性、缓存、降级与可追溯性
- 建立模型版本管理、在线学习和离线评估闭环
- 建立监控、告警、审计与质量门禁
- 为后续实盘接入预留标准接口

### 1.3 当前基线

Phase 5 以当前已完成模块为基线：

## 2. Phase 5 目标

### 2.1 总目标

将现有“分析型系统”升级为“可运行、可回归、可监控、可审计”的量化投顾闭环。

### 2.2 目标拆解

| 目标 | 说明 |
|------|------|
| 统一入口 | 一个入口完成分析、推荐、模拟和报告生成 |
| 统一数据契约 | 所有模块共享一致的字段、时间戳和来源信息 |
| 统一决策链路 | 规则引擎、ML 推荐、交易顾问共享同一决策上下文 |
| 统一质量门禁 | 数据质量、信号质量、回测质量、模型质量有门槛 |
| 统一运行保障 | 缓存、重试、降级、日志、指标、告警可配置 |

## 3. 设计原则

### 3.1 以当前实现为主线

Phase 5 不推翻现有模块，而是在现有能力之上做编排、治理和收口，避免重复建设。

### 3.2 规则与模型并存

规则信号、技术分析、基本面分析和 ML 推荐都保留，系统负责融合而不是互相替代。

### 3.3 先可控，再自动化

所有自动化动作必须先经过质量判断和风险约束，默认保持“建议优先、执行保守”的策略。

### 3.4 结果可追溯

每次分析输出都应包含：
- 输入数据版本
- 数据来源
- 模型版本
- 决策原因
- 风险参数

├── observability/           # 新增：日志、指标、告警
├── pipeline/                # 新增：分析流水线编排
└── risk/                    # 复用：风险控制

### 4.2 运行链路

```text
用户请求
  ↓
StockOrchestrator
  ↓
数据获取 / 缓存 / 质量校验
  ↓
技术指标 + 基本面 + 市场面 + 风险参数
  ↓
规则决策 + ML 推荐 + 交易顾问
  ↓
报告 / 仪表盘 / 模拟交易 / 审计记录
  ↓
监控指标 / 告警 / 回归验证
```

### 4.3 智能体接入层

Phase 5 需要将现有能力暴露为统一的智能体工具，而不是让模型直接拼接底层算法。

#### 工具边界

| 工具 | 作用 |
|------|------|
| `analyze_stock` | 个股分析主入口，输出技术面、风险与建议 |
| `recommend_by_risk` | 统一推荐入口，按风险偏好自动融合长线/短线推荐 |
| `recommend_long_term` | 长线推荐的显式入口 |
| `recommend_short_term` | 短线推荐的显式入口 |
| `get_market_overview` | 市场概览与板块轮动 |
| `get_risk_profile` | 风险配置与仓位门禁 |
| `run_backtest` | 策略回测与结果摘要 |
| `generate_stock_report` | 结构化分析报告 |

#### 统一输出信封

所有工具输出应尽量遵循同一结构：

```json
{
  "ok": true,
  "tool": "recommend_by_risk",
  "category": "recommendation",
  "data_source": "mock",
  "summary": "已生成风险匹配推荐列表。",
  "data": {},
  "meta": {
    "payload_version": "1.0",
    "generated_at": "2026-06-15T00:00:00Z"
  }
}
```

#### 自动路由原则

- 用户没有明确说长线或短线时，优先路由到 `recommend_by_risk`
- 用户明确提到“长线”“短线”时，才路由到对应专项工具
- 用户提到“风控”“仓位”“风险等级”时，优先路由到 `get_risk_profile`
- 用户提到“市场”“大盘”“行情”时，优先路由到 `get_market_overview`
- 用户提到“回测”时，优先路由到 `run_backtest`

## 5. 核心模块设计

### 5.1 统一编排模块

#### 功能概述

统一封装分析入口，负责串联数据获取、特征计算、规则分析、ML 推荐、风险控制和结果输出。

#### 当前实现状态

- 已新增 `core/orchestrator.py`
- 已实现 `StockOrchestrator`
- 已实现 `create_stock_orchestrator`
- 已支持自然语言路由入口 `route()`
- 已支持 `analyze / recommend / market_overview / risk_profile / backtest / report`
- 统一编排器通过现有工具注册表复用项目能力，不重复实现底层算法
- 已接入 `core/data/governance.py` 作为缓存与质量标记层

#### 核心接口

```python
class StockOrchestrator:
    """统一编排器"""

    def analyze(self, stock_code: str, risk_profile: str = "moderate") -> dict:
        """执行完整分析链路"""
        pass

    def recommend(self, stock_code: str, mode: str = "hybrid") -> dict:
        """输出推荐结果"""
        pass

    def run_pipeline(self, stock_code: str) -> dict:
        """执行标准化流水线"""
        pass
```

#### 设计要求

- 统一返回结构
- 支持规则和模型双通道
- 失败时可局部降级
- 保留分析上下文和追踪信息

### 5.2 数据治理模块

#### 功能概述

为真实数据链路增加缓存、校验、重试和降级能力，确保上层模块稳定工作。

#### 当前实现状态

- 已新增 `core/data/governance.py`
- 已实现 `CacheManager`
- 已实现 `DataQualityChecker`
- 已实现 `DataSnapshot`
- 已通过 `core/data/__init__.py` 对外导出

#### 核心能力

| 能力 | 说明 |
|------|------|
| 缓存 | 对行情、财务、市场数据做短期缓存 |
| 失效 | 按时间窗口或数据版本失效 |
| 校验 | 检查字段完整性、时间顺序、空值比例 |
| 降级 | 真实数据失败时回退到备用数据源或上次快照 |
| 追踪 | 标注数据来源、拉取时间、是否降级 |

#### 核心接口

```python
class DataSnapshot:
    """统一数据快照"""

class DataQualityChecker:
    """数据质量校验器"""

class CacheManager:
    """数据缓存管理器"""
```

#### 功能概述

把协作规划结果进一步执行为多步骤工作流，支持市场概览、风控、分析、推荐、回测和报告的顺序串联。

#### 当前状态

- 已新增协作规划草案
- 已定义协作计划 schema
- 已实现工作流执行器与节点级审计
- 已新增标准工作流模板草案

#### 核心能力

| 能力 | 说明 |
|------|------|
| 计划执行 | 将协作计划转成可执行节点 |
| 节点串联 | 支持按顺序或依赖执行工具 |
| 结果聚合 | 汇总节点结果并输出统一工作流结果 |
| 失败降级 | 支持任务失败时保留主结果 |
| 审计回放 | 记录工作流执行轨迹 |

#### 核心接口

```python
class WorkflowExecutor:
    """工作流执行器"""

    def execute(self, user_input: str, risk_profile: str = "moderate") -> dict:
        pass
```

#### 推荐工作流任务组

1. `市场概览 -> 风控 -> 个股分析 -> 报告`
2. `市场概览 -> 推荐 -> 回测 -> 报告`
3. `选股 -> 基本面 -> 风控 -> 报告`

#### 设计要求

- 优先复用现有业务工具
- 不改变现有单工具调用
- 结果必须有统一 schema
- 规划、执行、审计必须贯通

## 6. 业务场景设计

### 6.1 场景一：单股分析

用户输入股票代码后，系统返回：
- 当前趋势判断
- 规则信号
- 基本面评分
- ML 推荐
- 风险建议
- 目标价和止损价

### 6.2 场景二：批量筛选

系统基于市场和基本面条件筛选股票池，并生成候选列表和排序理由。

### 6.3 场景三：模拟交易验证

将推荐结果送入模拟交易环境，验证策略在给定周期内的收益和风险表现。

### 6.4 场景四：训练与发布

基于历史数据和回测结果训练模型，评估通过后发布新版本，失败则保留旧版本。

## 9. 质量门槛

### 9.1 数据质量

- 主流程数据必须可追溯
- 关键字段缺失时必须降级或中止
- 数据源失败必须有备用路径

### 9.2 模型质量

- 训练集和测试集指标必须记录
- 新版本模型不得低于基线阈值
- 发布后必须支持回滚

### 9.3 系统质量

- 核心分析接口必须稳定
- 输出结构必须兼容下游模块
- 失败必须返回可解释错误

---

## 10. 风险与限制

| 风险 | 影响 | 缓解方式 |
|------|------|----------|
| 数据源波动 | 分析结果不稳定 | 缓存、降级、重试 |
| 模型漂移 | 推荐质量下降 | 周期重训、评估门禁 |
| 规则与模型冲突 | 决策不一致 | 统一编排和权重策略 |
| 输出结构变更 | 下游兼容性受影响 | 版本化输出协议 |
| 过度自动化 | 产生错误决策风险 | 保留人工复核入口 |
