# 软件设计规格说明 (SDD)
# Phase 89: 日常使用测试规范化与最小对话入口说明

---

## 1. 背景

当前项目已经具备日常投产链路、回看链路、验收链路和自动回归门禁，整体更接近“可日常使用”的状态，而不是继续堆叠新功能的开发期。

但从实际使用角度看，当前入口仍然偏分散，用户需要自己判断：

- 今天应该先跑什么
- 跑完以后先看什么
- 哪些结果可以用于参考
- 项目是否存在真正的前端对话入口

Phase 89 的目标不是新增分析能力，而是把“日常使用测试”变成一个明确、可重复、低摩擦的使用协议，同时把对话式入口的现状说明清楚。

---

## 2. 总目标

把项目从“功能已具备”推进到“可稳定日用、可明确上手、可快速判断入口”的状态。

要求达到：

- 用户可以按照统一顺序进行日常使用测试
- 用户可以快速判断今天该跑哪些入口
- 用户可以清楚知道当前是否存在前端对话 AI 入口
- 不新增复杂编排、不引入新平台

---

## 3. 设计原则

### 3.1 先可用，再好看

优先保证用户能照着固定顺序跑起来，而不是先追求入口形态丰富。

### 3.2 入口清晰，少歧义

明确哪些入口用于健康检查、哪些用于日常决策、哪些用于回看、哪些用于验收。

### 3.3 对话入口先说明现状

如果没有真正的前端对话页，就不要包装成已经存在；应明确告知当前只有 CLI / HTTP API / 日常脚本入口。

### 3.4 不扩层

本阶段只做使用规范和最小入口说明，不新增新的分析层、调度层或前端框架层。

---

## 4. 范围

### 4.1 纳入范围

- 日常使用测试标准流程
- 默认阅读顺序
- 最小可执行入口说明
- 前端对话入口现状说明
- README / 任务清单 / 路线图更新

### 4.2 不纳入范围

- 新 Web 前端
- 新聊天 UI 框架
- 新调度系统
- 新模型或新分析层

---

## 5. 目标输出

Phase 89 重点输出以下使用协议：

- `daily_usage_runbook`
- `entry_order`
- `front_end_chat_status`
- `minimal_entry_path`
- `daily_test_checkpoints`

---

## 6. 规则约定

### 6.1 日常使用测试顺序

建议默认顺序为：

1. 数据健康
2. 日常决策
3. 回看摘要
4. 投产验收
5. 自动回归门禁

### 6.2 对话入口说明

当前项目没有独立的前端对话 AI 页面。

若希望做交互式使用，当前最接近入口的是：

- `tools/agent-api.py`
- `examples/api_demo.py`
- `core/api_server.py`

### 6.3 轻量化原则

日常使用测试只检查“是否能稳定执行”和“是否能稳定读懂结果”，不把本阶段变成持续调参或无限优化入口。
