## 1. 背景

Phase 40 到 Phase 50 已经把自选股日常链路做得比较完整了：

- `daily_run` 可以跑
- `review` 可以看
- `feedback` 可以记
- `feedback_insights` 可以看趋势
- `daily_watchlist_display.py` 也把终端摘要统一了一部分

但现在还缺一个真正面向“投产前验收”的最小检查入口。

也就是说，我们已经有很多块积木了，但还没有一个脚本能回答：

- 今天这套日常链路是不是完整的
- 关键文件有没有落下来
- 关键入口能不能继续运行
- 现在是否可以当作默认工作流来使用

因此，本阶段不增加新能力，而是把现有产物收成一个最小投产验收脚本。

---

## 2. 目标

### 2.1 总目标

让自选股日常链路可以被一键验收，输出一个明确的 `ok / degraded / failed` 结论。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 文件检查 | 确认 latest、run_status、feedback、pipeline 等关键产物是否存在 |
| 入口检查 | 确认 review 和 feedback_insights 还可运行 |
| 状态明确 | 给出 `ok / degraded / failed` 结论 |
| 轻量输出 | 终端直接可读，JSON 便于后续处理 |
| 不改主链路 | 不影响现有 daily run、review、反馈和洞察脚本 |

---

## 3. 范围

### 3.1 纳入范围

---

## 4. 设计原则

### 4.1 先验收，再继续

先确认链路完整，再考虑继续扩展。

### 4.2 用现有产物验收现有产物

验收只检查已有文件和已有入口，不引入新的业务语义。

### 4.3 结果简洁

验收入口要一眼能看懂，不输出过多噪声。

### 4.4 保持轻量

脚本应适合日常执行，不应依赖复杂环境或额外服务。

---

## 5. 输入输出约定

### 5.1 输入

建议支持：

- `--archive-dir`
- `--feedback-file`
- `--run-status`
- `--output-json`

### 5.2 输出

验收入口至少应输出：

- `status`
- `generated_at`
- `latest_json`
- `latest_md`
- `run_status`
- `feedback_file`
- `review_ok`
- `insights_ok`
- `summary`

### 5.3 状态定义

| 状态 | 含义 |
|------|------|
| `ok` | 关键文件存在，关键入口可运行 |
| `degraded` | 部分文件缺失，但主链路仍可用 |
| `failed` | 关键产物缺失或关键入口不可运行 |

---

## 6. 验收标准

- 可以检查 `latest.json`、`latest.md`、`run_status.json`、反馈 JSONL 是否存在
- 可以验证 `review` 和 `feedback_insights` 是否可运行
- 可以导出验收 JSON
- 不影响现有日常运行链路

1. 先把最小验收脚本做出来，只检查现有产物和现有入口
