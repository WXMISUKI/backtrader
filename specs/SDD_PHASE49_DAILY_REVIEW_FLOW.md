## 1. 背景

Phase 40 到 Phase 48 已经把自选股日常链路、留档、查看、反馈和反馈洞察都做出来了。

但真实使用时，用户往往不想记一串脚本，而是希望有一个更短的“回看流程”：

1. 跑完日常结果
2. 看最新留档
3. 看最近反馈趋势
4. 决定要不要顺手记一条反馈

如果这些步骤还是散落在多个命令里，日常使用成本就会继续停留在“能用，但不够顺手”。

因此，本阶段不增加新分析能力，而是把已有的日常查看动作收成一个轻量回看流程入口。

---

## 2. 目标

### 2.1 总目标

把自选股日常的“运行 -> 查看 -> 反馈 -> 洞察”收束成一个更短的回看流程。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 回看更短 | 一个脚本串起查看器和反馈洞察 |
| 不改主流程 | 不影响 `daily_run`、`feedback`、`feedback_insights` 的原始职责 |
| 默认适合日常 | 适合每天打开 latest 时直接扫一遍 |
| 复用现有脚本 | 只做编排，不重算结果 |
| 保持轻量 | 不引入新的分析或存储层 |

---

## 3. 范围

### 3.1 纳入范围

- 新增日常回看脚本
- 默认先看 latest 留档
- 默认展示最近反馈洞察
- 可选顺带展示原始 JSON
- 更新 README、任务清单和归档

### 3.2 不纳入范围

- 新的分析模型
- 新的前端界面
- 新数据库
- 新告警系统

---

## 4. 设计原则

### 4.1 只编排，不发明

回看流程只负责调用已有入口，不重复实现逻辑。

### 4.2 先看最新，再看趋势

默认优先看 `latest` 留档，然后看反馈洞察。

### 4.3 最短路径

把日常使用的心智成本压到最低，不让用户在目录里找入口。

### 4.4 保持可回退

回看流程失败时，用户仍然可以分别执行原始的查看、反馈、洞察脚本。

---

## 5. 输入输出约定

### 5.1 输入

建议支持：

- `--archive-dir`
- `--limit-lines`
- `--feedback-file`
- `--feedback-limit`
- `--min-samples`
- `--show-json`

### 5.2 输出

回看流程应至少输出：

- latest 留档摘要
- 最近反馈摘要
- 最近反馈洞察摘要
- 可选 JSON 细节

---

## 6. 验收标准

- 可以一条命令看最新留档和最近反馈趋势
- 不影响现有 daily run、viewer 和 feedback insights
- 能在 README 里清楚说明使用方式
- 语法和轻量运行验证通过
