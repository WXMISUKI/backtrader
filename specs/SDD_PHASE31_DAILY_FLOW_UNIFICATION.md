## 1. 背景

Phase 28、29、30 分别解决了：

- 看什么：自选股日常决策清单
- 信不信：自选股数据健康预检
- 是否持有：持仓上下文接入

但它们目前仍然是三个分开的入口。日常使用时，用户还需要自己决定先跑哪个、后跑哪个，心智上并不够收口。

本阶段的目标不是再增加能力，而是把这三步串成一个统一的日常入口，让用户每天只需要一个命令，就能得到：

- 数据健康预检
- 持仓上下文
- 最终决策清单

---

## 2. 目标

### 2.1 总目标

把自选股日常流程收束成一个统一入口，降低日常运行成本，提升“拿来就用”的程度。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 一次执行 | 一个脚本完成预检和决策输出 |
| 顺序固定 | 先健康预检，再持仓上下文，再决策清单 |
| 输出清晰 | 结果保持分段展示，避免过度堆叠 |
| 复用现有逻辑 | 不重写分析、不重写健康预检、不重写持仓解析 |
| 可导出 | 统一入口仍支持 JSON 导出 |

## 4. 设计原则

### 4.1 统一入口，不统一实现细节

入口统一，内部仍然调用现有脚本级能力，避免把稳定逻辑打散。

### 4.2 复用优先

统一入口只编排，不复制健康预检和清单决策的核心规则。

### 4.3 输出分层

先展示预检结果，再展示持仓上下文，再展示决策结果，保持阅读顺序自然。

### 4.4 保持脚本独立可用

`watchlist_data_health.py`、`daily_watchlist_decision.py` 仍保留，方便单独调试。

---

## 5. 输入输出约定

### 5.1 统一入口输入

建议支持：

- `--watchlist`
- `--portfolio`
- `--start-date`
- `--end-date`
- `--risk-profile`
- `--limit`
- `--output-json`

### 5.2 统一入口输出

建议分三段输出：

- 数据健康预检摘要
- 持仓上下文摘要
- 日常决策清单

统一 JSON 导出中应同时包含：

- `health`
- `portfolio`
- `decision`

---

## 6. 验收标准

- 一个命令可以跑完预检和决策
- 输出顺序固定且可读
- 现有单独脚本不失效
- 统一入口 JSON 中保留三类结果
- 不影响 Phase 28 / 29 / 30 的原有行为
