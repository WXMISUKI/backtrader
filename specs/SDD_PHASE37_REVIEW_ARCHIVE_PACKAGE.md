## 1. 背景

Phase 31 到 Phase 36 已经把自选股日常流程、日更报告、日报对比、周报模板和阶段报告模板都串起来了。

这意味着系统已经能不断产出高质量留档内容，但这些内容目前仍然分散在不同输出层里：

- 日更报告适合当天扫读
- 报告对比适合看两天差异
- 周报模板适合周末回看
- 阶段报告模板适合按月或按阶段复盘

对于真正需要分享、保存、交接或归档的场景，这些产物最好再收成一个统一的“复盘留档包”。

本阶段的目标不是新增新的分析层，而是把现有报告产物打包成一个稳定的留档包，方便：

- 保存为一份目录化的 JSON
- 直接导出为可分享的文本
- 形成一个完整的复盘入口

---

## 2. 目标

### 2.1 总目标

把当前的日报、周报、阶段报告和对比结果收成一个统一的复盘留档包，适合保存、分享和后续回看。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 一包聚合 | 统一收纳多种报告产物 |
| 便于留档 | 输出可直接保存和复用 |
| 便于分享 | 可作为一个完整交付物 |
| 轻量聚合 | 只打包已有结果，不重算分析 |
| 保持兼容 | 不破坏现有日报/周报/阶段报告流程 |

## 4. 设计原则

### 4.1 打包优先

核心目标是把现有报告打包成一个留档包，而不是再生成新的分析层。

### 4.2 结构稳定

留档包结构固定，便于长期保存和比对。

### 4.3 复用已有结果

留档包直接引用 `daily_report`、`daily_comparison`、`weekly_report`、`stage_report` 和 `portfolio_summary`。

### 4.4 可分享

留档包应当足够完整，适合复制给别人或放到文档系统中。

---

## 5. 输入输出约定

### 5.1 输入

继续沿用 Phase 36 的统一入口输入，并建议支持：

- `--archive-package`
- `--archive-name`

### 5.2 输出

建议新增 `archive_package` 对象，包含：

- `name`
- `generated_at`
- `summary`
- `daily_report`
- `daily_comparison`
- `weekly_report`
- `stage_report`
- `portfolio_summary`
- `report_text`

留档包文本建议包含：

- 复盘总览
- 当日摘要
- 变化对比
- 周报模板摘要
- 阶段报告摘要
- 持仓语境

---

## 6. 验收标准

- 可以把现有报告产物收成一个留档包
- 留档包适合保存和分享
- JSON 中保留完整结构
- 不影响原有日报、周报和阶段报告
- 不引入新的重型依赖

## 8. 下一步建议

1. 先把现有的日报、周报、阶段报告和对比结果聚成一个留档包
