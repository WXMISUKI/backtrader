## 1. 背景

Phase 43 到 Phase 45 已经把东方财富的 cookie 一致性、历史行情失败诊断和回归测试都收敛起来了。

但当前的实际体验仍然是：

- 预检脚本和日常脚本都能看到数据状态
- 但“健康状态”口径还分散在多个地方
- 你要判断问题时，还是需要自己拼接 `cookie_source`、`failure_kind`、`fallback_reason`、`data_source` 和质量结果

这会让项目继续停留在“诊断字段很多，但没有统一结论”的阶段。

因此，本阶段不再扩展东财接口，也不再加新的诊断类型，而是把现有信息统一收成一个稳定的健康摘要。

---

## 2. 目标

### 2.1 总目标

把东方财富相关的数据源状态统一收成一个可直接阅读的健康摘要，方便在日常预检和统一流程里快速判断问题归属。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 统一结论 | 预检和日常流程使用同一套健康摘要语义 |
| 读得更快 | 一眼能看出是正常、部分降级还是明显降级 |
| 保留细节 | 仍保留原始 `data_source`、`quality`、`reason` 和诊断字段 |
| 减少分叉 | 不在多个脚本里重复计算健康状态 |
| 不扩能力 | 不新增接口、不新增数据源、不新增网络逻辑 |

---

## 3. 范围

### 3.1 纳入范围

- 统一数据源健康摘要函数
- 复用到自选股数据健康预检
- 复用到日常统一流程
- 更新运行说明
- 更新任务清单和归档

### 3.2 不纳入范围

- 新增东财接口
- 新增重试或代理
- 新增自动刷新 cookie
- 新增健康评分模型

---

## 4. 设计原则

### 4.1 先统一口径，再谈优化

本阶段的关键不是“再多知道一点”，而是“先用同一种方式说清楚”。

### 4.2 复用现有诊断字段

直接复用已有的：

- `cookie_source`
- `data_source`
- `quality`
- `failure_kind`
- `fallback_reason`

### 4.3 保持轻量

摘要函数只做组合，不做新的推理和修复。

### 4.4 保持兼容

原来的输出字段保留，外部调用方不用改。

---

## 5. 输入输出约定

### 5.1 输入

健康摘要函数继续接收：

- `history`
- `fundamental`
- `stock_name`

### 5.2 输出

健康摘要应至少包含：

- `status`
- `health_score`
- `history_source`
- `fundamental_source`
- `history_quality`
- `fundamental_quality`
- `history_reason`
- `fundamental_reason`
- `summary`
- `flags`

### 5.3 状态定义

| 状态 | 含义 |
|------|------|
| `完全可用` | 关键链路均为真实可用数据 |
| `部分降级` | 至少一条链路降级，但仍可参考 |
| `明显降级` | 关键链路都已降级，不建议直接依赖 |

---

## 6. 验收标准

- `watchlist_data_health.py` 和 `daily_watchlist_pipeline.py` 使用同一套健康摘要语义
- 运行时能明确看到是完全可用、部分降级还是明显降级
- 原有诊断字段仍然保留
- README 能说明当前优先看哪一层摘要
- 不影响现有东财诊断和回归测试
