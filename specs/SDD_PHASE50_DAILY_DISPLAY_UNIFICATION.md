## 1. 背景

Phase 40 到 Phase 49 已经把自选股日常链路、反馈、洞察和回看流程都做出来了。

但现在几个入口的终端输出还是各写各的，虽然内容已经完整，但阅读节奏还不完全统一。

这会带来一个很实际的问题：

- 用户每次打开不同入口，看到的标题样式不完全一致
- 关键字段有时是键值对，有时是大段文字
- 回看和洞察的阅读体验还可以再收一点

因此，本阶段不再增加新能力，而是把现有日常入口的终端显示统一一下，让阅读更平整。

---

## 2. 目标

### 2.1 总目标

统一自选股日常几个入口的终端摘要格式，让每天查看时更一致、更省脑力。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 标题统一 | 入口标题、区块标题统一使用同一类样式 |
| 摘要统一 | 关键字段尽量用一致的键值输出 |
| 读取顺滑 | 日常回看、反馈洞察、留档摘要看起来更像一套东西 |
| 保持轻量 | 不改业务逻辑，只改显示层 |
| 保持兼容 | 原有脚本职责和输出内容不被破坏 |

---

## 3. 范围

### 3.1 纳入范围

- 抽出共享显示辅助工具
- 统一 `archive_viewer` 的摘要样式
- 统一 `feedback_insights` 的摘要样式
- 统一 `review` 的区块标题样式
- 更新 README、任务清单和归档

### 3.2 不纳入范围

- 不改业务数据结构
- 不改反馈写入
- 不改留档内容
- 不新增分析能力

---

## 4. 设计原则

### 4.1 只改显示，不改语义

输出内容不变，只让它们更像一套东西。

### 4.2 复用共享 helper

共享标题、键值对和 bullet 样式，避免每个脚本再写一遍。

### 4.3 保持克制

不引入复杂格式化框架，只做最小抽取。

### 4.4 面向日常阅读

样式目标是“每天扫一眼能看懂”，不是做报告模板竞赛。

---

## 5. 输入输出约定

### 5.1 输入

继续沿用现有入口参数，无需新增业务参数。

### 5.2 输出

统一后的输出应至少让以下部分更一致：

- 顶层标题
- 区块标题
- 关键字段键值对
- 列表样本

---

## 6. 验收标准

- `archive_viewer`、`feedback_insights`、`review` 的终端摘要样式更一致
- 现有命令仍可正常运行
- README 仍能清楚说明使用方式
- 不影响 daily run 和 feedback 写入
