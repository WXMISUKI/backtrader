## 1. 背景

Phase 60 到 Phase 67 已经把日常主链路收得很完整：

- `production_gate` 解决今天能不能参考
- `action_list` 解决今天先做什么
- `run_cadence` 解决今天跑到了哪一步
- `prompt_context` 解决这些材料怎么喂给智能体和 Skill

但日常使用时，用户仍然需要在回看时自己把这些内容重新组织一遍，才能快速判断今天到底该怎么看。

Phase 68 的目标，就是把回看入口和留档查看再收成一段统一的 `review_brief`，让用户打开时先看到短摘要，再继续看完整内容。

---

## 2. 总目标

建立一个最小可复用的回看摘要层，让系统在每天打开回看时直接回答：

- 今天先看哪一层
- 先处理门禁还是先看行动
- 反馈效果是否值得继续参考
- 这份回看是否适合快速扫一眼就结束

---

## 3. 核心原则

### 3.1 摘要先于正文

`review_brief` 不是新的报告。

它只是把已有材料压成一段更短、更适合日常打开的回看封面。

### 3.2 先门禁，再行动，再效果

回看时的阅读顺序应优先保持：

1. `production_gate`
2. `action_list`
3. `run_cadence`
4. `prompt_context`
5. `feedback_effects`

### 3.3 保持薄层

本阶段不新增模型，不新增评分，不新增调度，只收紧回看呈现。

### 3.4 终端简短，正文完整

回看入口先给短摘要和关键点，再继续输出完整留档、反馈洞察和反馈效果。

## 5. 回看摘要定义

建议把回看摘要拆成以下内容：

| 字段 | 含义 |
|---|---|
| `summary_text` | 面向终端的简短回看摘要 |
| `next_step` | 今天回看时建议先看什么 |
| `risk_note` | 回看时需要注意的风险提示 |
| `read_order` | 推荐阅读顺序 |
| `key_points` | 今日回看的关键点 |
| `metrics` | 反馈效果的轻量摘要 |
| `evidence` | 回看摘要来自哪些现有材料 |

---

## 6. 规则约定

### 6.1 `block` 语境

当门禁为 `block`：

- 回看摘要必须先提示先处理阻断
- 不能把 `review_now` 包装成强执行建议
- 重点应放在门禁原因和数据问题上

### 6.2 `warn` 语境

当门禁为 `warn`：

- 回看摘要应强调谨慎复核
- 先看行动清单和提示语境，再看反馈效果
- 不把降级结果当成强信号

### 6.3 `pass` 语境

当门禁为 `pass`：

- 可以按正常顺序复核
- 仍然保留风险提示
- 重点看行动是否合理、效果是否稳定

建议在 JSON 中新增：

```json
{
  "review_brief": {
    "status": "warn",
    "summary_text": "回看摘要：当前需要谨慎复核，先看门禁，再看行动和效果。",
    "next_step": "先复核行动清单和提示语境，再看反馈效果。",
    "risk_note": "存在降级，先复核再动作。",
```

## 9. 验收标准

- 日常回看入口能先看到 `review_brief`
- 留档查看入口能先看到 `review_brief`
- 回看摘要能区分 `pass / warn / block`
- 回看摘要能提供清晰的下一步建议
- 反馈效果仍然保留在完整正文里，不被摘要替代
