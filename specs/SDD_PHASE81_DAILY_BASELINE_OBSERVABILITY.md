## 1. 背景

Phase 78 到 Phase 80 已经把日常投产基线收成了一个很薄的入口：

- `status`
- `failed_stage`
- `failure_class`
- `repair_hint`
- `evidence_summary`

这已经足够回答“今天能不能跑通”“失败属于哪一类”“机器和脚本先看什么”。

但真实日常里，还缺一个更直接的问题：

> 最近几次基线是否反复卡在同一类问题上？

因此 Phase 81 只做一件事：把基线结果收成一个轻量观测层，用来识别高频失败、缺失项和 provider 可见性是否持续异常。

---

## 2. 总目标

建立一个最小的基线观测层，让系统每天都能快速回答：

- 最近几次运行最常见的失败类别是什么
- 哪些检查项最常缺失
- 历史 provider 可见性是否稳定
- 当前应该优先修哪一类问题

本阶段只做观测和排序，不引入新模型，不改主链路，不新增前端或调度系统。

---

## 3. 设计原则

### 3.1 保持薄层

只在 `daily_watchlist_production_baseline.py` 上增加观测摘要，不重新编排日常流程。

### 3.2 优先看趋势

单次失败仍然重要，但更重要的是最近 N 次是否持续出现同类问题。

### 3.3 证据可回溯

观测层要能直接关联到 `checks`、`evidence_summary` 和历史留档文件，便于回看。

### 3.4 不制造新概念

只复用现有字段，不发明新的分析模型或分级体系。

---

## 4. 范围

### 4.1 纳入范围

- 基线历史记录留档
- 最近 N 次运行摘要
- 失败类别统计
- 缺失检查项统计
- provider 可见性统计
- 高优先级修复顺序

### 4.2 不纳入范围

- 新策略
- 新模型
- 新前端
- 新调度系统
- 新数据源

---

## 5. 建议输出

建议在基线 JSON 中新增 `baseline_observation`，至少包含：

| 字段 | 含义 |
|---|---|
| `summary_text` | 一句话观测结论 |
| `recent_runs` | 最近 N 次运行摘要 |
| `failure_class_counts` | 失败类别计数 |
| `top_missing_checks` | 最常缺失的检查项 |
| `provider_visibility_rate` | provider 可见性比例 |
| `recommended_fix_order` | 推荐优先修复顺序 |
| `history_path` | 历史留档路径 |

---

## 6. 规则约定

### 6.1 `ready`

最近运行稳定，观测层应突出“保持现状并继续观察”。

### 6.2 `caution`

链路可跑，但应突出最近高频缺口和建议复核点。

### 6.3 `failed`

存在阻断时，观测层优先展示失败类别分布和修复顺序。

### 6.4 `blocked`

若阻断来自门禁层，观测层只能给出定位和修复提示，不输出强行动建议。

## 8. 验收标准

- 能看到最近 N 次基线运行趋势
- 能识别高频失败类别
- 能识别最常缺失的检查项
- 能看到 provider 可见性趋势
- 不破坏 `ready / caution / failed / blocked` 语义

---

## 9. 后续建议

Phase 81 完成后，优先观察高频故障闭环是否足够稳定，再决定是否继续细化修复优先级或扩展到别的日常薄层。
