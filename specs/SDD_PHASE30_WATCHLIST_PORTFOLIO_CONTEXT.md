## 1. 背景

Phase 28 的日常决策清单和 Phase 29 的数据健康预检已经把“看什么”和“信不信”解决得比较清楚了，但日常投研里最关键的一块还缺着：

- 这只股票我现在是不是已经持有
- 持仓占总资产多少
- 持仓是否需要继续关注、减仓或加仓

如果没有持仓上下文，日常清单会更像“候选列表”，而不是“可执行清单”。

因此，本阶段把最小持仓上下文接入日常清单，不做复杂组合管理，只做可读、可解释、可落地的持仓标记。

---

## 2. 目标

### 2.1 总目标

让日常清单在输出股票建议时，能同时看到持仓状态和仓位占用，帮助用户快速判断“现在是新开仓、继续持有，还是该减仓/关注风险”。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 持仓可见 | 每只股票明确显示是否已持有 |
| 仓位可见 | 明确显示单股仓位和组合仓位占比 |
| 输出轻量 | 只做读式接入，不做复杂交易引擎 |
| 配置简单 | 使用本地 JSON 配置持仓上下文 |
| 方便联调 | 日常清单脚本直接能消费持仓配置 |

## 4. 设计原则

### 4.1 以读为主

持仓上下文只用于解释和排序，不直接驱动交易执行。

### 4.2 复用现有仓位概念

优先沿用风险管理里的最大仓位和现有模拟交易的持仓表达，不引入新名词。

### 4.3 配置即事实

本阶段持仓上下文以本地配置为准，明确由用户维护，不尝试自动推断真实账户。

### 4.4 输出克制

只给出“已持有 / 未持有 / 仓位占用 / 关注动作”，不扩展成完整组合诊断。

---

## 5. 输入输出约定

### 5.1 持仓配置

建议使用 `config/portfolio.json`，每条持仓至少包含：

- `stock_code`
- `name`
- `size`
- `avg_cost`

可选字段：

- `market_price`
- `target_weight`
- `note`

### 5.2 输出字段

每只股票建议输出：

- `is_position`
- `position_size`
- `avg_cost`
- `market_price`
- `position_value`
- `position_weight`
- `position_label`
- `position_summary`

### 5.3 状态定义

| 状态 | 含义 |
|------|------|
| 已持有 | 该股票在持仓配置中存在 |
| 未持有 | 该股票不在持仓配置中 |
| 仓位过高 | 单股或总仓位已接近上限 |
| 仓位正常 | 当前持仓在可接受范围内 |

---

## 6. 验收标准

- 可以从 `config/portfolio.json` 读取持仓信息
- 日常清单可以显示每只股票是否已持有
- 日常清单可以显示仓位占用和持仓摘要
- 输出仍然保持轻量、可读
- 不影响 Phase 28/29 的数据健康和决策逻辑
