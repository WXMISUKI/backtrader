## 1. 背景

Phase 31 到 Phase 34 已经把自选股日常流程、日更摘要、日报格式化和日报对比都打通了。

这意味着每天的结果已经足够清楚，但周末复盘仍然需要把连续几天的日报再收成一份更高层的周报视图，才能更方便回答：

- 这一周整体是变好还是变差
- 哪些标的反复进入重点关注
- 哪些标的长期只是继续观察
- 持仓语境是否稳定
- 数据健康是否有持续波动

本阶段目标不是新增新逻辑，而是把多份日报收成一份稳定的周报模板，帮助做阶段复盘。

---

## 2. 目标

### 2.1 总目标

把连续的日报记录整理成一份可复用的周报模板，适合周末查看、保存和回顾。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 周视图 | 让用户一眼看懂这一周的变化 |
| 可复盘 | 周报内容便于长期留档 |
| 轻量聚合 | 只聚合现有日报，不重算分析结果 |
| 保持兼容 | 不破坏现有日报流程 |
| 模板化 | 便于后续每周重复使用 |

## 4. 设计原则

### 4.1 周报来自日报

周报只聚合已有日报，不单独定义新的分析口径。

### 4.2 保持模板稳定

周报段落顺序固定，便于每周持续对比。

### 4.3 复用现有结构

周报基于 `daily_report`、`daily_comparison` 和 `health` / `decision` 的累计结果生成。

### 4.4 不引入复杂状态

优先使用本地 JSON 数组作为输入，不引入数据库或后台服务。

---

## 5. 输入输出约定

### 5.1 输入

建议支持：

- `--weekly-from`
- `--weekly-to`
- `--weekly-reports`

其中 `weekly-reports` 可以是一个或多个日报 JSON 路径。

### 5.2 输出

建议新增 `weekly_report` 对象，包含：

- `period`
- `headline`
- `summary`
- `health_trend`
- `decision_trend`
- `portfolio_trend`
- `highlights`
- `report_text`

周报文本建议包含：

- 本周总览
- 健康趋势
- 决策趋势
- 持仓变化
- 本周重点标的
- 下周观察建议

---

## 6. 验收标准

- 可以把多份日报聚合成一份周报模板
- 周报能反映本周的总体变化趋势
- 周报文本适合留档
- JSON 中保留周报结构
- 不影响现有日报产物
