## 1. 背景

当前项目已经拥有：

- 统一决策入口
- 决策会话
- 用户反馈
- 任务协议
- 回放与统计

如果继续只停留在类方法和工具入口，虽然内部可用，但仍然不够“直接投产”。

为了尽快投入生产，本阶段只做一层最薄的 HTTP API 封装，不引入新的 Web 框架依赖。

---

## 2. 目标

### 2.1 总目标

提供一个可以快速启动的最小 HTTP API，让外部系统能直接调用统一决策入口、提交反馈和查看健康状态。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 最小依赖 | 使用标准库实现，不引入新框架 |
| 易启动 | 一条命令即可启动 |
| 易调用 | 提供统一 JSON 输入和输出 |
| 易上线 | 入口单一、逻辑简单 |
| 易调试 | 提供健康检查接口 |

---

## 3. 接口设计

### 3.1 `POST /decision`

统一决策入口。

请求体建议：

- `user_input`
- `risk_profile`

响应体应为统一决策结果。

### 3.2 `POST /feedback`

提交决策反馈。

请求体建议：

- `session_id`
- `workflow_id`
- `accepted`
- `rating`
- `reason`
- `correction`
- `comment`

### 3.3 `GET /health`

健康检查接口。

建议返回：

- `ok`
- `service`
- `status`
- `timestamp`

---

## 4. 设计原则

### 4.1 只做薄层

HTTP 层只负责请求转发和 JSON 编解码，不再引入业务逻辑。

### 4.2 统一输出

接口输出尽量复用现有统一决策结果。

### 4.3 快速上线

优先保证可启动、可调用、可调试。

---

## 5. 验收标准

- 可以启动最小 HTTP 服务
- 可以通过 `POST /decision` 获取统一决策结果
- 可以通过 `POST /feedback` 提交反馈
- 可以通过 `GET /health` 查看状态
- 不依赖额外 Web 框架
