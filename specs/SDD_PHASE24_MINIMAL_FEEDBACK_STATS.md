## 1. 背景

当前项目已经具备：

- 统一决策入口
- 标准工作流总入口
- 决策会话
- 最小反馈闭环
- 回放与统计能力

但在“快速投入生产、边用边改”的私用场景里，光有反馈写入还不够，还需要一个足够轻的入口来快速看到效果：

- 最近有多少次决策
- 采纳率大概是多少
- 最近的反馈主要是什么
- 这套推荐到底有没有开始被使用

因此，本阶段要补一个最小反馈统计入口，让真实测试能直接看到结果，而不是只停留在日志和文件里。

---

## 2. 目标

### 2.1 总目标

提供一个最小、只读、快速可用的反馈统计入口，让用户能够快速查看采纳率、反馈分布和最近样本。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 最小入口 | 只提供统计查询，不增加复杂看板 |
| 快速可见 | 一次请求即可看到采纳率和最近反馈 |
| 可联调 | 可以和现有 `/decision`、`/feedback` 联动 |
| 可复盘 | 统计结果可直接服务于后续优化 |
| 低成本 | 复用现有会话与反馈存储 |

---

## 3. 范围

### 3.1 纳入范围

- `GET /decision/stats`
- `StockOrchestrator.get_decision_session_stats`
- `examples/api_demo.py` 的统计查看能力
- README 中的最小统计示例

### 3.2 不纳入范围

- 复杂仪表盘
- 多维交互分析页
- 新数据库或数据仓库
- 复杂报表导出

---

## 4. 设计原则

### 4.1 只读优先

统计入口只负责读取和聚合，不写入额外业务状态。

### 4.2 复用现有会话

统计必须复用 `decision_sessions.jsonl` 和 `decision_feedback.jsonl`，避免引入新存储。

### 4.3 输出统一

统计入口也应保持与现有工具一致的统一输出结构，方便智能体和脚本直接消费。

### 4.4 先轻量，再扩展

先把采纳率、最近会话、最近反馈看清楚，再决定是否需要更细的统计维度。

---

## 5. 接口约定

### 5.1 `GET /decision/stats`

建议支持参数：

- `limit`：返回最近样本数量，默认 `20`

### 5.2 输出要求

输出至少应包含：

- `ok`
- `tool`
- `category`
- `data_source`
- `summary`
- `data`
- `meta`

其中 `data` 中至少应包含：

- `total_sessions`
- `accepted_count`
- `rejected_count`
- `accept_rate`
- `recent_sessions`
- `recent_feedback`
- `workflow_learning`

---

## 6. 验收标准

- `GET /decision/stats` 可以返回统计结果
- 可以看到采纳率和最近反馈样本
- 不影响现有 `/decision` 和 `/feedback`
- 示例脚本可以直接展示统计结果
- 本地无需新增部署依赖即可验证
