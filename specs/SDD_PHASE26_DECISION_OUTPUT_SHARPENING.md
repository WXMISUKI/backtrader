## 1. 背景

当前项目已经具备：

- 统一决策入口
- 标准工作流总入口
- 最小反馈闭环
- 最小反馈统计入口
- 最小反馈洞察入口

但在真实使用中，虽然系统已经能返回完整结构化结果，用户最常先看的仍然是同一件事：

- 这次的结论是什么
- 依据是什么
- 风险是什么
- 下一步动作是什么

为了更适合“快速投产、真实测试、边用边改”的私用场景，本阶段不再增加新入口，而是把统一决策输出再收紧一点，让结果更直接、更稳定、更适合人工阅读和智能体消费。

---

## 2. 目标

### 2.1 总目标

让统一决策输出的核心信息固定为四段式：

- 结论
- 依据
- 风险
- 下一步动作

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 更明确 | 统一把结论、依据、风险、下一步动作直接暴露出来 |
| 更一致 | `/decision`、运行时、工具调用和示例脚本保持一致 |
| 更易读 | 人工看结果时不必先翻很深的嵌套结构 |
| 向后兼容 | 保留原有 `decision` 结构和字段，不破坏现有调用方 |
| 低成本 | 只调整输出表达，不新增复杂业务逻辑 |

---

## 3. 范围

### 3.1 纳入范围

- `StockOrchestrator.answer_decision_request`
- `StockOrchestrator.answer_decision_summary`
- `StockAgentRuntime.answer_decision_summary`
- `ArkAgentClient` 的输出提示
- `examples/api_demo.py` 的展示方式
- `README_QUANT.md` 的使用说明

### 3.2 不纳入范围

- 新的路由规则
- 新的评分逻辑
- 新的数据库或存储层
- 新的 HTTP 接口
- 自动改写推荐结果

---

## 4. 设计原则

### 4.1 先结论，再展开

最前面暴露结论、依据、风险和下一步动作，降低阅读成本。

### 4.2 保留兼容

原有 `decision` 结构继续保留，避免影响既有脚本和调用方。

### 4.3 输出收口

统一输出不再鼓励读者从多个层级拼装答案，而是提供一份可直接使用的摘要。

### 4.4 不增加新复杂度

本阶段不修改路由、工作流或算法，只优化输出表达。

---

## 5. 接口约定

### 5.1 `POST /decision`

建议在原有返回值中新增：

- `decision_summary`
- `结论`
- `依据`
- `风险`
- `下一步动作`
- `conclusion`
- `basis`
- `risk`
- `next_action`

其中 `decision_summary` 为主摘要对象，建议包含：

- `结论`
- `依据`
- `风险`
- `下一步动作`
- `conclusion`
- `basis`
- `risk`
- `next_action`
- `confidence`
- `data_source`

### 5.2 `answer_decision_summary`

该入口应返回更适合直接阅读的简化结果，优先暴露：

- `结论`
- `依据`
- `风险`
- `下一步动作`

并保留必要的会话和治理信息。

### 5.3 输出要求

输出至少应保持：

- `ok`
- `tool`
- `scenario`
- `summary`
- `decision`
- `decision_summary`
- `data_source`
- `session_id`
- `workflow_id`
- `route_audit_id`
- `task_protocol`
- `governance`

---

## 6. 验收标准

- `POST /decision` 返回中能直接看到四段式摘要
- `decision_summary` 与 `decision` 保持一致
- `answer_decision_summary` 能直接返回更短、更清晰的结构
- 现有调用方式不需要改代码也能继续工作
- 示例和运行说明能说明新的输出方式

## 8. 下一步建议

1. 先把 `POST /decision` 的摘要字段收紧
2. 再让运行时与示例脚本优先展示四段式摘要
3. 最后把这一轮收口写入知识库和归档，作为后续统一输出的默认标准
