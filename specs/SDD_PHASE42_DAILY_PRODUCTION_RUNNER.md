## 1. 背景

Phase 40 已经把自选股统一流程稳定沉淀成按日留档产物，Phase 41 也补了一个固定查看入口。

现在链路里已经有：

- 数据健康预检
- 日常统一流程
- 自动归档
- latest 查看入口

但日常实际使用时，还缺一个更贴近“每天开工”的入口：

- 先判断今天能不能跑
- 再执行统一流程
- 最后把结果稳定留档
- 并给出一条清晰的状态结论

因此，本阶段不新增分析能力，而是把现有能力收束成一个一键运行守门入口。

---

## 2. 目标

### 2.1 总目标

把自选股日常流程收束成一个“先预检、再执行、后留档”的一键生产入口。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 一键执行 | 一个命令完成预检、执行、留档 |
| 运行守门 | 先判断配置和数据健康，再决定是否继续 |
| 状态明确 | 给出 `ok / degraded / failed` 级别结论 |
| 复用现有能力 | 不重写分析逻辑，不重造报告链路 |
| 默认留档 | 自动启用 Phase 40 的归档能力 |

---

## 3. 范围

### 3.1 纳入范围

- 新增一键运行脚本
- 复用数据健康预检
- 复用统一日常流程
- 复用自动归档与查看入口
- 生成运行状态文件

### 3.2 不纳入范围

- 新增分析模型

- 新增复杂调度系统
- 重构现有报告模板

---

## 4. 设计原则

### 4.1 先守门，再执行

如果配置或数据明显不可靠，入口要先把状态说清楚，再决定是否继续。

### 4.2 只编排，不重算

一键入口只负责把已有脚本串起来，不重复实现已有逻辑。

### 4.3 默认适合每天跑

入口默认应该适合日常使用，减少手动参数和路径判断。

### 4.4 结果可回看

执行后应默认沉淀到 Phase 40 的归档目录，并能被 Phase 41 直接查看。

---

## 5. 输入输出约定

### 5.1 输入

建议复用以下输入：

- `--watchlist`
- `--portfolio`
- `--start-date`
- `--end-date`
- `--risk-profile`
- `--archive-dir`
- `--limit`

### 5.2 输出

建议新增一键运行输出：

- `run_status`
- `preflight`
- `pipeline`
- `archive`
- `view`

并生成 `run_status.json`，至少包含：

- `status`
- `generated_at`
- `watchlist_path`
- `portfolio_path`
- `archive_dir`
- `pipeline_output`
- `latest_json`
- `latest_md`
- `summary`

### 5.3 状态定义

| 状态 | 含义 |
|------|------|
| `ok` | 预检通过，流程执行成功，归档成功 |
| `degraded` | 预检或数据存在降级，但流程仍可执行 |
| `failed` | 配置错误、关键预检失败或流程执行失败 |

---

## 6. 验收标准

- 可以用一个命令完成预检、执行和归档
- 能明确看到今天是 `ok`、`degraded` 还是 `failed`
- 能自动复用 Phase 40 的归档目录
- 能自动衔接 Phase 41 的查看入口
- 不影响现有统一流程和留档内容
