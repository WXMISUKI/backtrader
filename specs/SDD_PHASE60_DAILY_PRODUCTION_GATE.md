## 1. 背景

当前项目已经具备一条较完整的自选股日常链路：

- `daily_watchlist_flow.py` 负责默认日常流程
- `daily_watchlist_daily_run.py` 负责运行和归档
- `daily_watchlist_review.py` 负责回看
- `daily_watchlist_acceptance.py` 负责投产验收
- `daily_watchlist_diagnosis_bundle.py` 负责诊断收口

但现在仍缺少一个最终的“投产质量门禁”来回答：

> 今天这份输出到底能不能用于决策参考？

如果没有这个门禁，系统虽然能生成报告，但用户仍然需要人工判断数据是否降级、验收是否通过、建议是否应被采纳。Phase 60 的目标就是把这些材料汇总成统一、明确、可被智能体和脚本共同消费的门禁结果。

---

## 2. 总目标

建立一个最小可用的 `production_gate`，在每日流程结束后输出：

- 是否可以参考今天的结果
- 哪些动作允许
- 哪些动作需要禁止
- 门禁依据来自哪些现有材料

该门禁必须保持轻量，只做汇总和约束，不新增分析模型，不重写现有业务链路。

---

## 3. 关键原则

### 3.1 门禁不是单一规则

统一质量门禁不是某一个孤立评分，也不是单个 `health_score` 阈值。

它是一套生产协作协议，综合以下来源：

- 数据治理结果
- 健康摘要
- 诊断证据
- 日常决策分组
- 投产验收结果
- 持仓上下文
- 智能体输出约束

### 3.2 先约束，再建议

智能体、Skill、CLI 和后续 API 都应该先读取门禁结果，再决定输出建议的强度。

如果门禁为 `block`，系统只能输出诊断和观察建议，不能输出强行动建议。

### 3.3 保持薄层

Phase 60 不重算指标，不新增策略，不新增复杂模型。只把现有产物统一成明确门禁。

### 3.4 私人投产优先

门禁服务于私人日常使用，重点是快速判断和风险收口，不追求企业级审批流或复杂权限系统。

## 5. 门禁状态定义

| 状态 | 含义 | 允许动作 |
|---|---|---|
| `pass` | 数据和验收整体可用，可以参考日常决策 | 允许查看重点关注、持仓观察、普通观察 |
| `warn` | 存在部分降级或验收警告，只能谨慎参考 | 允许观察和复核，不建议重仓或新增强动作 |
| `block` | 核心数据不可用或验收失败，不建议用于交易决策 | 只允许诊断、修复数据、等待下一次运行 |

## 6. 输入材料

建议从现有产物中读取：

| 输入 | 来源 | 用途 |
|---|---|---|
| `daily_summary` | 日常流程或归档 | 判断整体状态和决策分布 |
| `diagnosis_counts` | 日常摘要 | 判断主要降级原因 |
| `diagnosis_evidence` | 诊断证据视图 | 展示门禁依据 |
| `acceptance` | 投产验收 JSON | 判断流程是否满足投产要求 |
| `health.items` | `latest.json` | 判断每只股票数据健康 |
| `daily_run_status` | 日常流程状态 | 判断运行是否失败或降级 |

---

## 7. 输出契约

建议在 JSON 中新增：

```json
{
  "production_gate": {
    "status": "pass",
    "summary": "今日数据和验收整体可用，可以参考日常决策。",
    "allowed_actions": ["review_now", "hold_watch", "wait"],
    "blocked_actions": [],
    "risk_level": "normal",
    "reasons": [],
    "evidence": {
      "daily_run_status": "ok",
      "acceptance_status": "ok",
      "diagnosis_counts": {},
      "degraded_count": 0,
      "blocked_count": 0
    }
  }
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `status` | `pass / warn / block` |
| `summary` | 面向用户的简短解释 |
| `allowed_actions` | 当前允许的行动类型 |
| `blocked_actions` | 当前禁止的行动类型 |
| `risk_level` | `normal / cautious / unsafe` |
| `reasons` | 触发警告或阻断的原因 |
| `evidence` | 结构化证据，供 CLI、API、智能体继续消费 |

## 8. 初始门禁规则

### 8.1 `block` 条件

满足任一条件即可进入 `block`：

- 默认日常流程执行失败
- 投产验收状态失败
- 所有或大部分自选股核心行情数据不可用
- `daily_summary.status` 为明显不可用或数据不足占比过高
- 诊断证据显示核心数据源整体不可用

### 8.2 `warn` 条件

满足任一条件可进入 `warn`：

- 部分股票数据降级
- 存在较多 `diagnosis_counts`
- 投产验收有警告但未失败
- 有重点关注股票，但其数据可信度不足
- 使用了缓存或备用数据，但仍可辅助观察

### 8.3 `pass` 条件

同时满足以下条件可进入 `pass`：

- 默认日常流程运行成功
- 投产验收通过
- 核心数据大部分真实可用
- 没有阻断级降级原因
- 重点关注股票具备足够数据可信度

## 9. 智能体与 Skill 协作方式

### 9.1 工具层

后续可以新增或复用一个轻量函数，例如：

```python
build_production_gate(
    daily_summary: dict,
    diagnosis_evidence: dict,
    acceptance: dict,
    health_items: list[dict],
    daily_run_status: str,
) -> dict
```

该函数只负责生成门禁结果，不负责重新分析股票。

### 9.2 Skill 层

相关 Skill 和工具入口在输出建议前应读取 `production_gate`：

- `stock-advisor`
- `stock-recommender`
- `trading-advisor`
- 日常自选股工作流
- 后续可能新增的 API 或智能体工具

### 9.3 智能体提示词层

提示词应明确约束：

- 门禁为 `pass` 时，可以正常解释重点关注和观察建议。
- 门禁为 `warn` 时，必须强调谨慎，只能给观察、复核和小心处理建议。
- 门禁为 `block` 时，不能给买入、加仓、强执行建议，只能解释问题和建议修复数据。

### 9.4 输出层

CLI、JSON、诊断包、后续 API 都应该展示同一份门禁结果，避免不同入口口径不一致。

## 11. 验收标准

- 默认日常流程 JSON 中包含 `production_gate`。
- 诊断收口包能展示同一份 `production_gate`。
- 门禁能区分 `pass / warn / block`。
- 门禁为 `block` 时，不应输出强行动建议。
- 门禁为 `warn` 时，输出必须体现谨慎和复核。
- 门禁结果有结构化证据，方便智能体和后续 API 消费。
