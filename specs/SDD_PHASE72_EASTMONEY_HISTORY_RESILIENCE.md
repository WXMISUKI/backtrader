## 1. 背景

Phase 71 已经把东方财富历史行情的会话路由修正到可配置、可诊断的状态，明确了当前 `quant` 环境里不只是 cookie 配置问题，而是历史接口在该环境下仍可能被远端直接断开。

当前链路已经能在失败时回退到 mock，但上层看到的仍然主要是 `data_source = mock`、`failure_kind = request`、`fallback_reason` 这类偏粗的信号。

这会带来一个实际问题：

- 数据层知道失败了
- 但日常健康摘要、联调输出和后续门禁还不够清楚地知道“失败发生在哪一层”“失败属于哪一类”“回退到底用了什么策略”

因此，本阶段不再继续猜 cookie、也不继续堆重试，而是把历史行情失败收束成更稳定的结构化语义，方便后续统一消费。

---

## 2. 目标

### 2.1 总目标

把东方财富历史行情失败收敛成一套稳定、可读、可回归的结构化失败语义，并显式输出回退策略。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 失败分层 | 区分请求层、响应层、质量层失败 |
| 失败编码 | 固定可回归的 `failure_code`，避免只剩泛化文本 |
| 回退显式 | 明确输出 `fallback_strategy`，让上层知道走的是哪种退路 |
| 下游可读 | 日常健康摘要和联调脚本可以直接读取这些字段 |
| 低侵入 | 不引入新数据源，不重构主请求逻辑 |
| 保持兼容 | 现有 `data_source / quality / reason / failure_kind` 继续保留 |

## 3. 范围

### 3.1 纳入范围

- 东方财富历史行情失败分类增强
- 治理快照补充结构化失败字段
- 联调脚本输出增强
- 日常健康摘要消费新字段
- 最小回归测试
- 运行说明、路线图和任务清单更新

### 3.2 不纳入范围

- 新增备用外部行情源
- 新增代理池或复杂重试
- 新增自动 cookie 刷新
- 重写治理层数据结构

---

## 4. 设计原则

### 4.1 先把失败说清楚

这一步不是为了让历史接口“更聪明”，而是为了让失败信号稳定、统一、可回归。

### 4.2 保留现有回退

仍然回退到 mock，但必须把回退策略和失败类别一起写清楚。

### 4.3 让下游少拼字段

联调脚本、健康摘要、日常诊断不要再自己猜“这是请求失败还是质量失败”，而是直接消费统一字段。

### 4.4 兼容现有链路

不改变默认调用方式，不改变成功路径的数据结构。

---

## 5. 输入输出约定

### 5.1 输入

继续沿用现有历史行情输入：

- `symbol`
- `start_date`
- `end_date`
- `period`
- `adjust`

### 5.2 输出

在治理快照和联调结果中，建议补充以下字段：

- `failure_kind`
- `failure_stage`
- `failure_code`
- `fallback_strategy`
- `fallback_reason`

### 5.3 状态定义

| 字段 | 含义 |
|------|------|
| `failure_kind` | 兼容旧语义的大类，如 `request / response / quality` |
| `failure_stage` | 更具体的失败发生层，如 `request / response / quality / unknown` |
| `failure_code` | 可回归的稳定编码，如 `remote_disconnected / response_format / missing_columns` |
| `fallback_strategy` | 当前采用的回退策略，如 `offline_history_snapshot` |
| `fallback_reason` | 当前回退的具体文本原因 |

---

## 6. 验收标准

- 历史行情失败时可以稳定看到 `failure_kind / failure_stage / failure_code / fallback_strategy`
- 联调脚本能直接展示这些字段
- 日常健康摘要能消费这些字段，而不必自己重新拼接
- 现有成功路径不受影响
- 不引入新的外部依赖和新的数据源假设
