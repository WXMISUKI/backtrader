## 1. 背景

Phase 63 已经把日常结果收成 `action_list`，让系统能回答“今天先做什么”。

但在真实日常使用里，仍然存在一个小而关键的问题：

> 用户知道该看哪一组，但还不够直观看到每一组今天该怎么处理。

所以 Phase 64 不再新增入口，而是把行动清单里的提示语句做得更直接一些，让每条行动同时具备：

- 为什么放到这里
- 今天先做什么
- 需要先看什么证据

---

## 2. 总目标

在不改变行动分类规则的前提下，给 `action_list` 补充更清晰的行动提示语句，让终端和 JSON 输出都更像一个可执行的日常待办清单，而不是只读诊断结果。

---

## 3. 核心原则

### 3.1 不改主分类

`review_now / hold_watch / wait / skip_due_to_data / diagnose` 的分类规则保持不变。

### 3.2 提示语句要短

提示语句必须短、直、能执行，避免再写成一段长解释。

### 3.3 提示语句要和行动一致

提示语句不能和 action 本身冲突，不能把 `skip_due_to_data` 写成强行动建议。

---

## 4. 范围

### 4.1 纳入范围

- 为 `action_list` 增加 `action_hint`
- 让 `summary_text` 直接体现更明确的行动提示
- 在默认流程和留档查看入口里展示更明确的提示语句
- 在投产验收里校验提示语句是否存在
- 更新运行说明、任务清单和归档

### 4.2 不纳入范围

- 不改行动分类规则
- 不引入排序模型
- 不引入自动交易
- 不做复杂归因

---

## 5. 提示语句定义

| 行动 | 提示语句方向 |
|---|---|
| `review_now` | 今天优先复核，先看关键证据和最近变化 |
| `hold_watch` | 先检查仓位、成本和风险线，确认持有是否仍合理 |
| `wait` | 先不动作，等待下一轮数据或更明确触发条件 |
| `skip_due_to_data` | 先跳过本轮判断，优先排查数据或可信度问题 |
| `diagnose` | 先排查门禁阻断原因，再决定是否恢复后续判断 |

---

## 6. 规则约定

### 6.1 依据行动自动生成提示

提示语句应由行动类别、门禁状态、持仓状态和健康状态共同生成。

### 6.2 保持终端可读

终端输出要能在前三到五条里直接读懂“今天先做什么”。

### 6.3 兼容 JSON 消费

`action_hint` 必须作为结构化字段保留，供 CLI、验收、留档查看和后续智能体消费。

---

## 7. 输出契约

建议在 `action_list.items` 中新增：

```json
{
  "action_list": {
    "items": [
      {
        "stock_code": "000001",
        "action": "skip_due_to_data",
        "reason": "分组 数据不足；...",
        "action_hint": "先跳过本轮判断，优先排查数据或可信度问题"
      }
    ],
    "summary_text": "..."
  }
}
```

## 9. 验收标准

- `action_list` 中每条行动都带有明确 `action_hint`
- 默认流程和留档查看能展示新的提示语句
- 验收可以确认提示语句存在
- 不改变原有行动分类和门禁判断
