## 1. 背景

当前项目已经具备个股分析、基本面分析、市场分析、推荐、回测、风控和交易顾问等能力。下一阶段的重点不应再是单点算法堆叠，而是把这些能力组织成“智能体可调用、可追踪、可复用、可演进”的业务体系。

为了避免后续开发依赖零散经验，本阶段需要把可学习的外部开源项目、可复用的工程模式、以及本项目的决策边界沉淀为知识库。

---

## 2. 目标

### 2.1 总目标

构建一份面向后续开发的知识库和路标文档，帮助项目在以下方面保持一致：

- 智能体如何编排现有能力
- 哪些能力应该继续做成工具
- 哪些能力应该上升为统一入口
- 哪些外部开源项目值得学习
- 哪些设计不应盲目照搬

### 2.2 交付物

| 交付物 | 说明 |
|------|------|
| 知识库文档 | 汇总外部项目、可学习模式、落地建议 |
| 路标清单 | 为 Phase 5 后续任务提供优先级 |
| 约束原则 | 明确哪些能力必须保留统一输出与降级机制 |

---

## 3. 范围

### 3.1 纳入范围

- 量化研究平台的工程组织方式
- 多智能体金融分析的角色拆分方式
- 数据服务、缓存、回测和评估闭环
- 报告、审计、风控和统一输出格式
- 技能生态可复用的通用能力

### 3.2 不纳入范围

- 具体交易策略的收益承诺
- 未验证的数据源直接替换当前实现
- 为了“更智能”而增加无边界的工具数量
- 用单一模型替代现有算法与规则体系

## 5. 外部参考对象

### 5.1 Qlib / Qlib-Server

参考价值：

- 学习研究到生产的分层方式
- 学习数据集、特征、实验与评估的标准化
- 学习在线数据服务的设计

可借鉴点：

- 数据服务与研究流程分离
- 统一的数据集和实验入口
- 支持离线研究与在线消费两种模式

### 5.2 vn.py

参考价值：

- 学习模块化交易框架的边界划分
- 学习事件驱动与网关适配的组织方式
- 学习面向实盘扩展的工程结构

可借鉴点：

- Gateway / Engine / Strategy 的职责分离
- 插件化接入不同券商或交易通道
- 以框架能力服务策略开发，而不是把策略和系统强耦合

### 5.3 TradingAgents

参考价值：

- 学习多角色智能体的任务拆分
- 学习研究型 agent 的分工协作
- 学习“调研、研判、总结、风控”分层表达

可借鉴点：

- 使用不同角色处理不同决策维度
- 让 agent 负责组织和总结，而不是替代全部底层计算
- 风险与结论分开输出

### 5.4 FinRobot

参考价值：

- 学习金融专用 agent 平台的整体思路
- 学习 LLM、量化、风险、报告的统一编排
- 学习如何把研究过程产品化

可借鉴点：

- 多技术路线并存
- 面向金融分析的统一输出格式
- 报告和分析链路可追溯

## 6. 本项目落地方向

### 6.1 必须保留的核心能力

- `analyze_stock`
- `recommend_by_risk`
- `get_market_overview`
- `get_risk_profile`
- `run_backtest`
- `generate_stock_report`

### 6.2 推荐新增的中层能力

- `StockOrchestrator`
- 数据质量检查器
- 数据缓存层
- 模型注册与版本记录
- 审计日志与统一追踪 ID

### 6.3 推荐的智能体行为

- 用户未指定策略时，默认走风险匹配推荐
- 用户指定“长线/短线”时，再进入专项工具
- 工具返回必须带来源和降级标记
- 回测、推荐、报告必须共享同一输出信封

---

## 7. 质量门槛

| 门槛 | 说明 |
|------|------|
| 统一输出 | 所有工具尽量返回统一信封 |
| 来源可见 | 输出必须标记 real / mock / config 等来源 |
| 降级可见 | 降级路径不能隐藏 |
| 可回溯 | 每次分析保留时间戳与参数 |
| 可复验 | 回测和推荐可重跑 |

---

## 8. 验收标准

- 知识库中能明确说明项目下一阶段方向
- 知识库中能明确说明可借鉴的外部项目及用途
- 知识库中能明确说明哪些能力应复用、哪些应新增
- 知识库中能明确说明智能体接入原则和输出格式原则
