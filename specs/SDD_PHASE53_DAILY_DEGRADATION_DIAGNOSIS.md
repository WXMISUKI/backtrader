## 1. 背景

Phase 46 已经把数据健康摘要统一起来，Phase 52 也把默认日常工作流收口完成了。

但在实际使用里，当前的 `degraded / 需要谨慎` 还偏“结果态”，不够“原因态”。

也就是说，用户能看出今天不算完全健康，但还不够容易快速判断：

- 是历史行情链路的问题
- 还是基本面链路的问题
- 还是 mock 回退、异常、质量不稳定导致的

因此，本阶段不再新增分析能力，而是把现有健康摘要再往前收一层，输出可读、可统计、可传递的降级原因。

---

## 2. 目标

### 2.1 总目标

让自选股日常结果在出现降级时，不只告诉用户“降级了”，还告诉用户“为什么降级”。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 原因透明 | 能分出历史、基本面、mock、异常、质量不稳等原因 |
| 层级清晰 | 共享健康摘要输出统一诊断字段 |
| 总览可读 | 日常总览里能看到原因统计 |
| 保持轻量 | 不新增新数据源，不新增新业务逻辑 |
| 适合日常 | 能直接服务 daily_run / review / acceptance / flow |

## 4. 设计原则

### 4.1 结果态向原因态前移

保留 `status`，但要补上 `diagnosis`，让日常判断更直接。

### 4.2 复用现有健康语义

优先复用 `history_source`、`fundamental_source`、`quality`、`is_degraded`，不要重做一套判断逻辑。

### 4.3 总览要能扫

总览输出应允许一眼看出今天主要是哪些原因在触发降级。

### 4.4 不扩边界

只做诊断表达收口，不去修数据源、不去加告警、不去做自动重试策略。

---

## 5. 输出约定

### 5.1 共享健康摘要建议输出

- `status`
- `health_score`
- `history_source`
- `fundamental_source`
- `history_quality`
- `fundamental_quality`
- `history_reason`
- `fundamental_reason`
- `diagnosis`
- `flags`

### 5.2 诊断字段建议输出

`diagnosis` 至少应包含：

- `primary_cause`
- `primary_label`
- `severity`
- `cause_tags`
- `summary`
- `details`

### 5.3 日常总览建议输出

- `diagnosis_counts`
- `降级原因` 汇总行

---

## 6. 验收标准

- 每只股票的健康摘要里能看到诊断字段
- 日常总览能看到降级原因统计
- 仍然保留原有健康分和状态语义
- 不影响默认工作流入口和验收入口
