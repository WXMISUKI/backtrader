# 软件设计规格说明 (SDD)
# Phase 87: 自动回归门禁与一键验证闭环

---

## 1. 背景

Phase 83 到 Phase 86 已经把关键门禁、主契约、provider 可见性和量产环境门禁收束到了日常主链路。

但当前仍缺一个更薄的“自动回归门禁”入口，用来把最关键的日常验证步骤串成一键执行、可分类、可回写的闭环。

Phase 87 只做一件事：把现有的 `daily_run -> acceptance -> baseline` 收成一个自动回归门禁，不再新增业务分析层。

---

## 2. 总目标

把日常验证从“分别跑多个入口”推进成“一个命令完成关键回归检查”。

要求达到：

- 可以一键执行关键回归步骤
- 可以把结果分类成 `pass / warn / block`
- 可以把回归结果回写到 `run_status` 和 `latest.json`
- 不改变现有业务语义和主链路结构

---

## 3. 设计原则

### 3.1 先验证，再判定

按照 `daily_run -> acceptance -> baseline` 的顺序执行，前一步失败时后续步骤可以跳过，不做无意义扩散。

### 3.2 只做薄门禁

不新建分析层，不新建模型，不新建编排平台，只包一层最薄的验证入口。

### 3.3 结果回写

门禁结果要回写到现有运行产物，方便后续验收和回看读取。

### 3.4 可验

必须能通过最小测试验证状态分类和 artifact 回写行为。

---

## 4. 范围

### 4.1 纳入范围

- 自动回归门禁入口
- 关键步骤顺序执行
- 结果分类与摘要
- 回写 `run_status` / `latest.json`
- 最小回归测试
- 归档与路线图同步

### 4.2 不纳入范围

- 新业务分析
- 新数据源
- 新模型
- 新调度平台

---

## 5. 目标输出

自动回归门禁应稳定输出：

- `status`
- `summary_text`
- `next_step`
- `step_order`
- `step_records`
- `blocked_steps`
- `warning_steps`
- `skipped_steps`

---

## 6. 规则约定

### 6.1 `daily_run`

先执行 `daily_watchlist_daily_run.py`，它是整个回归门禁的第一步。

### 6.2 `acceptance`

在 `daily_run` 可用时再执行 `daily_watchlist_acceptance.py`，用于确认日常产物和验证闭环。

### 6.3 `baseline`

在前两步都不阻断时再执行 `daily_watchlist_production_baseline.py`，用于确认最终基线状态。

### 6.4 回写语义

回归结果应回写到 `run_status`，若 `latest.json` 已存在也应同步回写。

### 6.5 语义不变

回归门禁只能影响验证判定，不应篡改已有的业务结果语义。

## 8. 验收标准

- 可以一键跑完日常关键回归步骤
- 结果能清楚区分 `pass / warn / block`
- 回归结果能写回现有产物
- 关键失败不会被误判为通过
- 不破坏 `ready / caution / failed / blocked` 语义
