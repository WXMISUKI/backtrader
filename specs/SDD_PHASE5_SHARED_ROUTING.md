## 1. 背景

当前系统已经具备：

- 智能体客户端路由
- 编排器自然语言路由
- 统一工具输出
- 数据治理和缓存

但智能体客户端和编排器内部仍存在两套近似的路由逻辑。

如果这两套逻辑继续分叉，就会出现：

- 同一句话在不同入口路由到不同工具
- 参数默认值不一致
- 新意图需要改两处
- 路由原因无法统一解释

因此需要把意图解析提升为共享模块。

---

## 2. 目标

### 2.1 总目标

让智能体客户端和 StockOrchestrator 复用同一套意图解析、默认参数和优先级规则。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 单一来源 | 路由规则只维护一处 |
| 一致性 | 智能体与编排器选择相同工具 |
| 可解释 | 路由结果可返回 intent / reason / confidence |
| 可扩展 | 新意图只需扩展共享模块 |

## 3. 范围

### 3.1 纳入范围

### 3.2 不纳入范围

---

## 4. 设计原则

### 4.1 共享优先

路由逻辑必须放在共享模块中，避免重复实现。

### 4.2 结果复用

共享模块输出的结构应同时适用于：

- 工具调用前的意图判断
- 编排器执行前的动作决定
- 调试和审计

### 4.3 兼容旧入口

旧的 `route()` 和 `ask()` 不需要改调用方式，只需内部复用共享模块。

---

## 5. 核心设计

建议共享模块输出：

```python
{
    "intent": "backtest",
    "tool": "run_backtest",
    "confidence": 0.98,
    "reason": "命中回测关键词",
    "arguments": {...}
}
```

---

## 6. 验收标准

- 智能体和编排器复用同一解析逻辑
- 相同输入在两个入口得到一致工具选择
- 路由结果可直接用于调试和日志
- 新增意图时无需修改两处逻辑

---
