## 1. 背景

Phase 31 到 Phase 39 已经把自选股统一入口逐步收敛成一条稳定的日常流程：

- 日更摘要
- 日更报告
- 日报对比
- 周报模板
- 阶段报告模板
- 复盘留档包

目前的输出主要还是以终端打印和单次 JSON 导出为主，适合验证和调试，但还没有形成真正适合日常沉淀的稳定产物流：

- 每天的结果没有固定落盘目录
- 历次运行之间没有统一的目录结构
- 没有便于快速回看的 `latest` 指针
- 没有轻量、稳定的 Markdown 交付版

因此，本阶段不再增加新的分析层，而是把现有输出固化成可持续使用的日常运行产物。

---

## 2. 目标

### 2.1 总目标

把现有统一入口的输出稳定落成“按日归档 + 最近可读 + 可导出 Markdown”的日常产物流。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 固定落盘 | 每次运行都能按日期沉淀结果 |
| 便于回看 | 通过 `latest` 快速定位最近一次结果 |
| 保留原结构 | 继续复用现有 JSON 输出内容 |
| 增加可读版 | 提供稳定的 Markdown 交付文件 |
| 轻量实现 | 不改分析判断，不扩展新报告层 |

## 4. 设计原则

### 4.1 只固化现有成果

本阶段只保存已经算出来的结果，不再引入新的判断逻辑。

### 4.2 产物结构稳定

日常产物需要稳定、可预期，避免今天一个样、明天一个样。

### 4.3 按日归档优先

每次运行都应当能落入日期目录，方便后续对比、回溯和清理。

### 4.4 最近结果易访问

通过 `latest.json` 和 `latest.md` 快速查看最近一次运行结果。

### 4.5 Markdown 只做可读交付

Markdown 的目标是方便人读，不是替代 JSON 的完整结构。

---

## 5. 输入输出约定

### 5.1 输入

继续沿用 Phase 39 的统一入口输入，并新增：

- `--archive-dir`
- `--export-markdown`
- `--write-latest`

### 5.2 输出

本阶段建议增加以下落盘产物：

- `archive-dir/YYYYMMDD/*.json`
- `archive-dir/YYYYMMDD/*.md`
- `archive-dir/latest.json`
- `archive-dir/latest.md`

其中：

- `archive-dir` 为归档根目录
- `YYYYMMDD` 为运行日期目录
- `latest` 为最近一次运行的快捷入口

### 5.3 JSON 内容

JSON 仍然保留现有字段：

- `daily_summary`
- `daily_report`
- `daily_comparison`
- `weekly_report`
- `stage_report`
- `archive_package`

### 5.4 Markdown 内容

Markdown 建议至少包含：

- 运行元信息
- 日更报告
- 日报对比
- 周报模板
- 阶段报告模板
- 复盘留档包摘要

---

## 6. 验收标准

- 运行脚本后能在指定目录看到按日归档结果
- 运行脚本后能生成可读的 Markdown 文件
- 启用 `--write-latest` 后能更新最近结果文件
- 现有 JSON 结构保持兼容
- 不影响既有分析与报告判断
