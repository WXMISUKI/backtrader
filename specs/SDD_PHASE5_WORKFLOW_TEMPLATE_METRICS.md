## 1. 背景

标准工作流模板已经接入协作规划与工作流执行，但如果没有统计和优先级优化，模板层很快会出现两个问题：

- 只能知道“有模板”，不知道“哪个模板更常用、更稳定”
- 当多个模板同时可命中时，仍然缺少统一、可解释的排序依据

因此需要把模板层从“静态目录”推进到“可观测、可排序、可回放”的运行态组件。

---

## 2. 目标

### 2.1 总目标

让标准工作流模板具备命中统计、优先级裁决和可回放分析能力，支撑后续模板迭代。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 命中统计 | 记录模板命中次数、来源和分布 |
| 优先级裁决 | 多模板候选时按统一评分选择 |
| 回放分析 | 可回看最近命中的模板与原因 |
| 可解释性 | 模板选择原因与评分可见 |
| 可扩展 | 新模板接入后自动纳入统计 |

---

## 3. 范围

### 3.1 纳入范围

- 模板命中日志
- 模板命中统计
- 模板选择评分
- 模板结果元数据
- 模板查看工具

### 3.2 不纳入范围

- 外部监控平台接入
- 分布式指标系统
- 复杂机器学习排序模型

---

## 4. 设计原则

### 4.1 轻量优先

优先采用 JSONL 记录和本地聚合，不引入额外基础设施。

### 4.2 可解释优先

评分可以复杂，但输出必须简明、可读、可审计。

### 4.3 与业务解耦

模板统计只负责记录和分析，不替代业务执行逻辑。

---

## 5. 核心设计

### 5.1 模板评分

每个模板在选择阶段应综合以下信号：

- 模板基础优先级
- 路由意图是否命中
- 路由候选是否与模板意图匹配
- 模板关键词是否命中
- 目标工具是否与模板任务工具匹配

### 5.2 模板命中记录

每次命中模板时，应记录：

- `template_id`
- `template_name`
- `selected_by`
- `template_score`
- `objective`
- `route`
- `mode`
- `task_count`

### 5.3 模板统计输出

建议统计结果包含：

- 总命中次数
- 各模板命中次数
- 各选择来源分布
- 最近命中记录

---

## 6. 对外接口

### 6.1 模板统计工具

建议新增：

```python
get_workflow_template_stats(limit: int = 20) -> dict
```

### 6.2 模板目录工具

模板目录仍由 `list_workflow_templates` 提供，用于查看当前可复用模板。

---

## 7. 验收标准

- 模板命中后可记录统计数据
- 多模板可命中时有稳定排序结果
- 模板选择原因包含评分信息
- 智能体可查询模板统计
- 现有工作流执行链路不被破坏
