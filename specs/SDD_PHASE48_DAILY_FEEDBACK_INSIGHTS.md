## 1. 背景

Phase 47 已经把自选股日常反馈闭环补出来了。

现在项目已经能够记录：

- 哪次日常结果被记了反馈
- 哪只股票被采纳、忽略、继续观察或认为错误
- 这些反馈会追加到本地 JSONL

但如果没有一个只读统计入口，反馈就还停留在“记下来了”，并没有变成“下一步可以直接看的趋势”。

因此，本阶段不继续扩大反馈写入能力，而是把反馈记录收成一个轻量统计洞察入口。

---

## 2. 目标

### 2.1 总目标

让自选股日常反馈能够被快速查看、粗略统计和趋势化阅读。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 轻量洞察 | 一眼看到最近反馈总量和分布 |
| 统计直观 | 能看到采纳、观察、忽略、错误的大致比例 |
| 便于回看 | 能看最近几条反馈样本 |
| 不改主链路 | 不影响现有反馈写入和日常运行 |
| 复用存储 | 继续读取现有 JSONL 反馈文件 |

---

## 3. 范围

### 3.1 纳入范围

- 新增日常反馈洞察脚本
- 统计最近反馈总数和动作分布
- 统计按股票代码的反馈频次
- 可选输出 JSON
- 更新 README、任务清单和归档

### 3.2 不纳入范围

- 新数据库
- 复杂归因模型
- 自动交易
- 前端图表
- 反馈标签体系重构

---

## 4. 设计原则

### 4.1 只读优先

本阶段只读反馈，不更改反馈写入语义。

### 4.2 简单聚合

先把数量、比例、最近样本看清楚，不做复杂分析。

### 4.3 复用现有文件

直接读取 `daily_watchlist_feedback.jsonl`，保持实现最小。

### 4.4 适合日常扫一眼

输出要短、稳定、便于每天开工前查看。

---

## 5. 输入输出约定

### 5.1 输入

建议支持以下输入：

- `--feedback-file`
- `--limit`
- `--min-samples`
- `--output-json`

### 5.2 输出

反馈洞察应至少包含：

- `total_feedback`
- `accepted_count`
- `rejected_count`
- `watching_count`
- `wrong_count`
- `recent_samples`
- `stock_counts`
- `summary`

---

## 6. 验收标准

- 可以读取现有反馈 JSONL
- 可以看到最近反馈总量和动作分布
- 可以看到最近样本和高频股票
- 不影响现有 feedback 写入脚本
- 不影响 daily run 和 archive viewer
