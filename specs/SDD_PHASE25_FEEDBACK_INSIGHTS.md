## 1. 背景

当前项目已经具备：

- 统一决策入口
- 最小反馈闭环
- 最小反馈统计入口
- 回放与学习统计

但在真实测试中，仅看“总采纳率”还不够。为了更快判断下一步该优化什么，还需要一个更直接的洞察入口：

- 哪些意图最值得优化
- 哪些工具更容易被采纳
- 哪些路径是高价值路径
- 哪些路径应该优先回收或改进

因此，本阶段要把统计进一步收敛成“反馈洞察”入口，让后续迭代能直接面向高价值优化点。

---

## 2. 目标

### 2.1 总目标

提供一个最小、只读、可解释的反馈洞察入口，用于识别高价值路径和优先优化对象。

### 2.2 具体目标

| 目标 | 说明 |
|------|------|
| 可解释 | 直接告诉用户哪些意图/工具值得优先优化 |
| 可收敛 | 把反馈数据变成可执行的优化目标 |
| 可复用 | 复用现有会话、反馈和学习统计 |
| 低成本 | 只做只读洞察，不新增复杂平台 |
| 快迭代 | 为下一轮路由和模板优化提供依据 |

---

## 3. 范围

### 3.1 纳入范围

- `GET /decision/insights`
- `StockOrchestrator.get_decision_feedback_insights`
- `examples/api_demo.py` 的洞察查看能力
- README 中的最小洞察示例

### 3.2 不纳入范围

- 自动修改路由规则
- 复杂 BI 看板
- 多维报表导出
- 新数据库或训练系统

---

## 4. 设计原则

### 4.1 只读优先

洞察入口只负责读取、聚合和解释，不自动改系统行为。

### 4.2 从统计到决策

统计只是结果，洞察要把结果变成“下一步该优化什么”的建议。

### 4.3 样本门槛

洞察结果要带样本量提示，避免样本太少时误导优化方向。

### 4.4 先最小，再收敛

先做意图、工具、场景三类最小聚合，再决定是否细分到更复杂的维度。

---

## 5. 接口约定

### 5.1 `GET /decision/insights`

建议支持参数：

- `limit`：返回最近样本数量，默认 `20`
- `min_samples`：洞察最小样本门槛，默认 `2`

### 5.2 输出要求

输出至少应包含：

- `ok`
- `tool`
- `category`
- `data_source`
- `summary`
- `data`
- `meta`

其中 `data` 中至少应包含：

- `total_feedback`
- `accept_rate`
- `average_rating`
- `by_intent`
- `by_tool`
- `optimization_targets`
- `high_value_paths`
- `optimization_notes`

---

## 6. 验收标准

- `GET /decision/insights` 可以返回反馈洞察
- 可以看出优先优化的意图和工具
- 可以区分高价值路径与待优化路径
- 不影响现有 `/decision`、`/feedback`、`/decision/stats`
- 本地无需新增部署依赖即可验证
